## 0. Spec grounding (§5.2)

The opus BLUEPRINT node already discovers enumerable cross-file sites, so per-file research is redundant for small/medium leaves — even an 8-site EDIT passed FLOOR in 1 attempt / 3 nodes. The real trigger is **SCALE**: too many files/too-large a diff for one implement turn, OR **non-enumerable fan-out** (dynamic dispatch, string-keyed sites, reflection) the blueprint can't surface. So: blueprint emits a size estimate; executor branches FLOOR vs WAVES; both bounded by the same node budget; thresholds tunable.

## 1. Structured size emission (shared schema — also serves todo 86b2f019 Bridge manifest)

### 1.1 What the blueprint node emits

Append a **single fenced ```json block** at the END of the blueprint file (`.collab/leaf-blueprints/<leaf.id>.md`). The prose blueprint is unchanged above it; the block is the machine-readable tail. (A fenced block in the .md is preferred over a sidecar: one artifact, already committed by the blueprint node, already read by downstream nodes, and trivially greppable by the Bridge.)

Augment `buildNodePrompt('blueprint', …)` to instruct the node to finish with EXACTLY:

````
```json
{ "schemaVersion": 1, "estimatedFiles": <int>, "estimatedTasks": <int>,
  "nonEnumerableFanout": <bool>,
  "filesToCreate": [<path>...], "filesToEdit": [<path>...],
  "tasks": [ { "id": "<slug>", "files": [<path>...], "description": "<one line>" } ] }
```
````

### 1.2 Schema (TypeScript, exported for reuse by 86b2f019)

```ts
export interface LeafSizeManifest {
  schemaVersion: number;            // start 1
  estimatedFiles: number;
  estimatedTasks: number;
  nonEnumerableFanout: boolean;
  filesToCreate: string[];
  filesToEdit: string[];
  tasks: Array<{ id: string; files: string[]; description: string }>;
}
```

**Shared-schema note (86b2f019).** This is the *single source of truth* for "what files/tasks a leaf touches." The Bridge file-manifest reads the SAME fenced block from the SAME blueprint artifact — `filesToCreate`/`filesToEdit`/`tasks[].files` are the manifest. Keep the schema additive-only (bump `schemaVersion`, never repurpose a field) so both consumers stay compatible. Export `LeafSizeManifest` + the parser from leaf-executor (or a tiny shared module) so the Bridge imports rather than re-implements.

### 1.3 Robust parse — `parseSizeManifest(blueprintText, sizeText)` (FAIL-SAFE = FLOOR)

Non-deterministic node output ⇒ parse defensively. The blueprint node writes the file; the executor reads it back (it already knows `blueprintPath(leaf)` and the worktree `cwd`). Pass BOTH the file contents and the node's returned `text` (the node may echo the block in either).

```
function parseSizeManifest(...sources: (string|undefined)[]): LeafSizeManifest | null
  for each source:
    extract LAST ```json ... ``` fence (regex /```json\s*([\s\S]*?)```/g, take last match)
    JSON.parse it inside try/catch
    validate: estimatedFiles/estimatedTasks are finite numbers >= 0,
              nonEnumerableFanout is boolean,
              tasks is an array (coerce missing arrays to [])
    on success → return normalized manifest
  return null    // NOTHING parsed
```

**Fail-safe contract:** `parseSizeManifest(...) === null` ⇒ executor treats the leaf as **FLOOR** (the proven default). A garbled estimate must NEVER force the more expensive WAVES path, and must never crash the run. This mirrors `parseVerdict`'s fail-closed posture. The node budget remains the deterministic backstop if the estimate is wrong in the *other* direction (a too-big leaf mis-classified small still parks BLOCKED on budget, never runs forever).

## 2. The SIZE GATE in `runLeaf` (insertion point)

Thresholds as named, tunable consts beside `ATTEMPT_CAP`/`NODE_BUDGET`:

```ts
export const FILE_THRESHOLD = 4;   // estimatedFiles <= this ⇒ floor-eligible
export const TASK_THRESHOLD = 6;   // estimatedTasks <= this ⇒ floor-eligible
```

Decision helper (pure, unit-testable):

```ts
export function shouldUseFloor(m: LeafSizeManifest | null): boolean {
  if (!m) return true;                       // unparseable ⇒ FLOOR (fail-safe)
  return m.estimatedFiles <= FILE_THRESHOLD
      && m.estimatedTasks <= TASK_THRESHOLD
      && !m.nonEnumerableFanout;
}
```

**Exact insertion** — inside the attempt loop, immediately AFTER the existing blueprint node + its rate/budget checks (current lines 335–337), BEFORE the implement node (line 340):

```
const bp = await runNode('blueprint', buildSpec('blueprint', cwd));
if (bp.rateLimited) return pausedResult('blueprint', bp);
if (!checkBudget()) return parkBlocked('node-budget-exhausted');

// --- P5 SIZE GATE (new) ---
const manifestText = await deps.readBlueprint?.(cwd, leaf).catch(() => undefined);
const manifest = parseSizeManifest(manifestText, bp.text);
if (!shouldUseFloor(manifest)) {
  const wavesResult = await runWaves(manifest!, { cwd, ... });  // §3
  if (wavesResult) return wavesResult;     // paused/blocked short-circuit
  // waves completed all files clean → FALL THROUGH to the REVIEW node below
} else {
  // FLOOR — UNCHANGED implement node:
  const impl = await runNode('implement', buildSpec('implement', cwd));
  if (impl.rateLimited) return pausedResult('implement', impl);
  if (!checkBudget()) return parkBlocked('node-budget-exhausted');
}

// REVIEW node + PASS/merge/complete + FAIL→next-attempt: UNCHANGED (lines 344–384)
```

Both branches converge on the **same leaf REVIEW node** and the same PASS/merge/complete/FAIL machinery — so the attempt/retry loop, the rate-cap pause, and the gate funnel are untouched. The ONLY structural change is *which* nodes run between blueprint and review.

`deps.readBlueprint` is a new injected seam (default: read `path.join(cwd, blueprintPath(leaf))` via fs; tests inject the manifest text directly). Optional `?.` keeps the floor working even if unwired.

## 3. The WAVES path — `runWaves(manifest, ctx)`

Lives in a small helper `src/services/leaf-waves.ts` (keeps `runLeaf` readable) OR an inner closure in leaf-executor. It receives the run-state closures by injection so the budget/pause/ledger semantics are IDENTICAL to the floor: `{ runNode, checkBudget, parkBlocked, pausedResult, buildWaveSpec, cwd }`.

Returns `LeafRunResult | null`:
- `null` ⇒ all files implemented + verified clean → caller falls through to the leaf REVIEW node.
- a `LeafRunResult` ⇒ a terminal short-circuit (paused on rate cap, or blocked on budget/stuck) the caller returns as-is.

### 3.1 Topological waves over `manifest.tasks`

1. Build a per-file work set from `tasks[].files` ∪ `filesToCreate` ∪ `filesToEdit`.
2. **Research wave** — one research node per task (`tasks[]`); these have no inter-dependencies so they MAY be ordered/parallelized (v1: sequential to keep budget accounting simple and deterministic; parallelism is an additive follow-up — note in code). Each research node writes findings into the blueprint dir for the implement node to read.
3. **Implement wave** — one implement node per file (group tasks by file). Topological order = files with no create-deps first; v1 may treat files as independent (the blueprint already sequenced enumerable sites) and iterate in `filesToCreate` then `filesToEdit` order.
4. **Verify wave** — one verify node per file: runs `tsc -p tsconfig.json` under the **PROJECT** config (NOT a standalone/temp tsconfig — see the known tsc-prompt follow-up; the verify prompt must `cd` to repo root and run the project build, so cross-file types resolve).
5. **Per-file fix loop** — on a verify failure, run a fix node scoped to that file; re-verify. Track `previousErrors` per file; **same error signature twice = stuck** ⇒ `parkBlocked('waves-file-stuck')`. (Matches §5.1 "same-error-twice = stuck".)
6. **Wave-level tsc gate** — after all per-file fixes settle, one final project-wide `tsc -p tsconfig.json` verify node. Must be clean before returning `null` (→ leaf REVIEW). A failure here ⇒ `parkBlocked('waves-tsc-gate-failed')`.

### 3.2 Node profiles for wave kinds

Extend `LeafNodeKind` to a superset for the ledger, OR keep `LeafNodeKind` as-is and add a parallel `WaveNodeKind`. Recommended: widen the kind union and `NODE_PROFILE`:

```ts
export type LeafNodeKind =
  | 'blueprint' | 'implement' | 'review'        // floor (unchanged)
  | 'research'  | 'wimplement' | 'verify' | 'fix'; // waves

NODE_PROFILE = {
  ...existing,
  research:   { model: 'opus',   allowedTools: 'Read Grep Glob Bash' },        // read-only
  wimplement: { model: 'sonnet', allowedTools: 'Read Edit Grep Glob Bash' },   // read+edit
  verify:     { model: 'sonnet', allowedTools: 'Read Grep Glob Bash' },        // read + bash-tsc (Bash is RO-by-convention; verify runs tsc)
  fix:        { model: 'sonnet', allowedTools: 'Read Edit Grep Glob Bash' },   // read+edit
};
```

(Reuse `implement` profile for `wimplement` if preferring not to widen — but a distinct kind gives cleaner ledger rollups for the cost/tiering work already in this epic. Keep `'implement'` reserved for the floor so floor ledger rows are unchanged.)

### 3.3 Building each NodeSpec

A `buildWaveSpec(kind, cwd, target)` mirrors the existing `buildSpec` but takes a per-file/per-task `target` interpolated into a kind-specific prompt (new `buildWavePrompt(kind, leaf, target)`):
- research: "Investigate task `<id>` (`<description>`), files `<files>`. Read-only. Write findings to the blueprint dir."
- wimplement: "Implement file `<file>` per the blueprint + research notes. Read/Edit only."
- verify: "Run `tsc -p tsconfig.json` from the repo root. Report the FIRST error for `<file>`, or `TSC: CLEAN`."
- fix: "Fix the tsc errors in `<file>`: <errors>. Read/Edit only."

Every spec sets `permissionMode:'bypassPermissions'`, `leafId`, `epicId` exactly as `buildSpec` does.

## 4. Ceilings preserved (CRITICAL)

- **Node budget (20):** EVERY wave node goes through the SAME `runNode` wrapper, which does `state.nodesSpent += 1` BEFORE the spawn. So research/wimplement/verify/fix all increment the master counter. After EACH wave node, the waves code calls `checkBudget()`; on `false` it returns `parkBlocked('node-budget-exhausted')`. Budget check fires at the SAME granularity as the floor: once per node, right after the spawn. Over budget ⇒ BLOCKED immediately, mid-wave, regardless of how many files remain.
- **Rate-cap pause:** every wave node checks `res.rateLimited` and returns `pausedResult(kind, res)` immediately — same as floor. `state.nodesSpent` is carried; the daemon resumes (and re-enters from blueprint, re-deriving the manifest — acceptable, the blueprint node is idempotent on a fresh worktree).
- **Attempt cap (2):** waves run ENTIRELY INSIDE one attempt iteration. A failed leaf REVIEW (after waves) falls through to the existing `if (isLastAttempt) return parkBlocked('attempt-cap-exhausted')` / else next fresh attempt. The whole waves block re-runs fresh on attempt 2 (fresh worktree, fresh blueprint, fresh manifest). No surgical reuse — that's §5.1/P6.
- **Fresh worktree:** unchanged; waves operate in the same `cwd = wt.path` already cut `fresh:true` off the epic tip.

## 8. Risks

- **R1 (highest): FLOOR drift.** The floor MUST stay byte-identical. Mitigation: lift the existing 3 implement lines verbatim into the `else`; do not refactor them. Cases (a)/(d) + the full P2 suite are the canary. If any P2 test changes, STOP.
- **R2: attempt/retry interaction.** Waves run fully inside one attempt; a failed review re-runs the whole waves block fresh on attempt 2 (fresh worktree). This does NOT complicate the attempt loop's *structure* — but it does mean attempt 2 can be expensive (re-runs research). Acceptable per spec (fresh-every-time, no surgical reuse until §5.1). Node budget is the backstop against a second full waves run blowing up. *Flag:* a big-leaf double-waves run could exhaust the 20-node budget on attempt 2 → BLOCKED rather than a clean retry; this is the intended deterministic backstop, not a bug, but worth surfacing in stats.
- **R3: tsc under project config.** Verify MUST run `tsc -p tsconfig.json` from repo root (not a standalone tsc on one file) or cross-file types won't resolve and verify will false-fail. This is the known tsc-prompt follow-up — bake the project-config invocation into the verify/fix prompts.
- **R4: manifest mis-estimate.** A too-big leaf estimated small runs FLOOR and may fail review → falls to attempt 2 (still bounded). A too-small leaf estimated big wastes nodes but is bounded by budget. Both fail safely; neither corrupts state.
- **R5: shared-schema coupling with 86b2f019.** Keep `LeafSizeManifest` additive-only (version it) so the Bridge consumer never breaks on a field change.
