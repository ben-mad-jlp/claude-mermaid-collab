## File 1 — `src/services/__tests__/cdp-input.test.ts` (NEW)

The module under test (`src/services/cdp-input.ts`) is produced by sibling task
`cdp-input-module`. It is **import-free and pure-ish**: every dispatch fn takes
`client: any` first and calls `client.Input.dispatchMouseEvent` / `dispatchKeyEvent`.
`canvasToPage` is a pure sync function. No `vi.mock` needed — just a hand-rolled mock client.

### Mock client pattern
```ts
function makeClient() {
  return { Input: { dispatchMouseEvent: vi.fn(), dispatchKeyEvent: vi.fn() } };
}
```
Assert with `expect(client.Input.dispatchKeyEvent).toHaveBeenNthCalledWith(n, {...})` and
`...).toHaveBeenCalledTimes(k)`. All dispatch fns are `async` → `await` them.

### Param-identity table — assert these EXACT objects (the acceptance bar)
Lifted byte-for-byte from `src/mcp/tools/browser.ts` (verified in sibling research
`cdp-input-module.md`). Key subtleties to encode as assertions:

| fn | call(s) — exact param object |
|---|---|
| `pressKey(client, 'a')` | call 1: `{ type: 'keyDown', key: 'a' }`; call 2: `{ type: 'keyUp', key: 'a' }`. **No `text`, no `code`.** dispatchKeyEvent called 2×. |
| `typeText(client, 'hi')` | per char, BOTH `text` AND `key` set, no `code`: keyDown `{ type:'keyDown', text:'h', key:'h' }`, keyUp `{ type:'keyUp', text:'h', key:'h' }`, then same for `'i'`. dispatchKeyEvent called 4× for "hi". |
| `moveMouse(client, 10, 20)` | `{ type: 'mouseMoved', x: 10, y: 20 }`. **No `button` field** (hover). |
| `dragMouse(client, 1, 2, 3, 4)` | 3 calls: `{ type:'mousePressed', x:1, y:2, button:'left', clickCount:1 }` → `{ type:'mouseMoved', x:3, y:4, button:'left' }` (button but **no clickCount**) → `{ type:'mouseReleased', x:3, y:4, button:'left', clickCount:1 }`. |
| `clickMouse(client, 5, 6)` (defaults) | `{ type:'mousePressed', x:5, y:6, button:'left', clickCount:1 }` → `{ type:'mouseReleased', x:5, y:6, button:'left', clickCount:1 }`. Also test explicit `clickMouse(client,5,6,'right',2)` → button `'right'`, clickCount `2`. |
| `scrollMouse(client, 7, 8, 0, -120)` | `{ type:'mouseWheel', x:7, y:8, deltaX:0, deltaY:-120 }`. |
| `dispatchKey(client, 'keyDown', { key:'Enter', code:'Enter', modifiers:2 })` | `{ type:'keyDown', key:'Enter', code:'Enter', modifiers:2 }` (i.e. `{ type, ...opts }`). Test a `'keyUp'` variant too. |
| `dispatchMousePress(client, 'mousedown', 9, 9)` | maps `mousedown→mousePressed`: `{ type:'mousePressed', x:9, y:9, button:'left', clickCount:1 }`. `'mouseup'→mouseReleased`: `{ type:'mouseReleased', x:9, y:9, button:'left', clickCount:1 }`. |

> NOTE on exact-match vs subset: `toHaveBeenCalledWith` does a deep-equality match on the
> WHOLE arg object, so it WILL catch an extra `text`/`code` the impl shouldn't emit
> (e.g. guards `pressKey` against accidentally adding `code`). This is the param-identity
> guarantee — prefer it over `objectContaining`. For `dispatchKey`, if the impl spreads
> `opts` and an undefined key sneaks in (`{type, key, text: undefined}`), deep-equal treats
> `{a:1}` vs `{a:1, b:undefined}` as **equal** in Vitest, so passing only the defined keys in
> the test's expected object is fine.

### `canvasToPage` math cases (pure, no client)
Formula: `x = (nx*deviceWidth)/(pageScaleFactor||1)`,
`y = offsetTop + (ny*deviceHeight)/(pageScaleFactor||1)`.

| meta | nx, ny | expected |
|---|---|---|
| `{offsetTop:0, pageScaleFactor:1, deviceWidth:1000, deviceHeight:800}` | 0.5, 0.5 | `{x:500, y:400}` |
| `{offsetTop:0, pageScaleFactor:2, deviceWidth:1000, deviceHeight:800}` | 0.5, 0.5 | `{x:250, y:200}` (zoom 2 halves) |
| `{offsetTop:50, pageScaleFactor:1, deviceWidth:1000, deviceHeight:800}` | 0, 0 | `{x:0, y:50}` (offsetTop adds to y only) |
| `{offsetTop:50, pageScaleFactor:2, deviceWidth:1000, deviceHeight:800}` | 1, 1 | `{x:500, y:450}` (=50 + 800/2) |
| `{offsetTop:0, pageScaleFactor:0, deviceWidth:100, deviceHeight:100}` | 1, 1 | `{x:100, y:100}` — the `||1` guard: psf 0 → divide by 1, NOT NaN/Infinity. **Include this guard case.** |

Assert with `toEqual({x, y})`. Values are exact (no float fuzz at these inputs); if a case
yields a repeating decimal, use `toBeCloseTo` — but the table above is all-integer.

### Suggested describe structure
`describe('cdp-input dispatch helpers')` with an `it` per fn (param-identity), plus
`describe('canvasToPage')` with the 5 math cases. ~9–11 `it`s total.

## File 2 — extend `src/__tests__/websocket-handler.test.ts`

### How to drive an inbound message
`handler.handleMessage(ws, message)` — signature (handler.ts:162):
`handleMessage(ws: ServerWebSocket<{subscriptions:Set<string>}>, message: string): void`.
The 2nd arg is a **JSON string** (it `JSON.parse`s internally). The whole body is wrapped in
`try/catch` (handler.ts:163 / catch at ~259) that only `console.error`s — so a throw inside
never propagates; the no-op test therefore can't rely on try/catch alone, but the
`if (this.browserInputHandler)` guard is what guarantees the genuine no-op.

The existing test file NEVER calls `handleMessage` (it only tests `broadcast*`). The
`createMockWS()` helper (lines 23–42) already builds a usable `ws` with
`data.subscriptions: Set`. Reuse it — `handleConnection` is optional for handleMessage
(handleMessage doesn't require the ws be in `connections`), but calling it is harmless and
matches the other tests' setup.

### The contract being tested (from `ws-browser-input.md` §3b)
Impl adds to `WebSocketHandler`:
- field `private browserInputHandler: ((msg:{session:string;event:any})=>void)|null = null;`
- setter `setBrowserInputHandler(cb)` 
- branch in `handleMessage`: `else if (data.type === 'browser_input') { if (this.browserInputHandler) this.browserInputHandler({ session: data.session, event: data.event }); }`

### Cases to add (new `describe('browser_input message')` block)
1. **Forwarded with `{session, event}` when a handler is registered.**
   ```ts
   const spy = vi.fn();
   handler.setBrowserInputHandler(spy);
   const ws = createMockWS();
   const event = { kind: 'click', nx: 0.5, ny: 0.5, button: 'left' };
   handler.handleMessage(ws, JSON.stringify({ type: 'browser_input', session: 's1', event }));
   expect(spy).toHaveBeenCalledTimes(1);
   expect(spy).toHaveBeenCalledWith({ session: 's1', event });
   ```
   (Optionally a 2nd assertion with a `keydown` event variant to show passthrough of any
   `event` shape — the handler forwards `data.event` opaquely, no narrowing.)

2. **Silent no-op when NO handler set (no throw).**
   ```ts
   const handler2 = new WebSocketHandler();         // fresh, no setBrowserInputHandler
   const ws = createMockWS();
   expect(() => handler2.handleMessage(
     ws, JSON.stringify({ type: 'browser_input', session: 's1', event: { kind:'click', nx:0, ny:0 } })
   )).not.toThrow();
   ```
   `beforeEach` already does `new WebSocketHandler()` per test, so the top-level `handler`
   has no handler registered unless a test sets it — case 2 can just use `handler` directly
   instead of a fresh instance. Either is fine; using the per-test `handler` is simplest.

3. (optional, defensive) **handler not invoked for an unrelated message type** — e.g. send a
   `{type:'connected', diagramCount:0}` after registering the spy and assert
   `spy` NOT called. Cheap guard that the new `else if` branch is correctly gated on
   `data.type === 'browser_input'`.

### Type note for the test
`setBrowserInputHandler` / `handleMessage` are real methods after impl lands, so no `as any`
needed. The message object passed to `JSON.stringify` is a plain literal (not typed as
`WSMessage`), so no cast required there either. If TS complains that `setBrowserInputHandler`
doesn't exist (e.g. running tests before the impl task lands), that's expected — this `tests`
task depends on `ws-browser-input` + `cdp-input-module` having landed. Do NOT stub the
methods; let the real impl satisfy them.
