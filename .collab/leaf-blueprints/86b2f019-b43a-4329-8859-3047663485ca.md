## 1. BACKEND — persist the blueprint per attempt

### 1a. New injected dep (keeps the state machine unit-testable)
Add ONE optional dep to `LeafExecutorDeps` (mirrors `readBlueprint`/`recordNode` — optional `?.` so the floor still runs if unwired):

```ts
/** Persist the just-written blueprint as a durable collab document and link it to
 *  the leaf todo (per ATTEMPT, so failed attempts survive). Best-effort: a throw
 *  must NEVER break the run. Returns the created doc id (telemetry only). */
persistBlueprint?: (input: {
  project: string;          // TRACKING project (where the todo lives) — NOT the worktree
  leaf: Todo;
  attempt: number;          // 1-based; included in doc name + body so attempts are distinct
  manifest: LeafSizeManifest;
  blueprintMd: string;      // the full .md text (prose + trailing json fence)
}) => Promise<string | undefined>;
```

### 1b. Call site (in `runLeaf`)
Immediately AFTER the blueprint node returns AND the manifest is derived — i.e. right after
`const manifest = parseSizeManifest(manifestText, bp.text);` (~line 609), BEFORE the size-gate
branch. Persist ONLY when we actually have the .md text and a parsed manifest:

```ts
if (manifestText && manifest) {
  try {
    await deps.persistBlueprint?.({
      project, leaf, attempt: state.attempt, manifest, blueprintMd: manifestText,
    });
  } catch { /* persistence is durable-telemetry — never break the run */ }
}
```

This fires on EVERY attempt (cap 2) → failed attempts get their own persisted doc, satisfying
"per ATTEMPT (so failed attempts survive, not just the accepted one)".

### 1c. Real wiring in `makeLeafExecutorDeps`
The executor runs against the worktree, but `createDocument`/`updateTodo` act on the TRACKING
`project`. Implement the dep using the real server APIs already found:
- `createDocument(name, content)` → `src/services/document-manager.ts`, returns the doc `id` (string).
  - `name`: `Leaf blueprint — <leaf.id.slice(0,8)> attempt <attempt>` (distinct per attempt).
  - `content`: the full `blueprintMd` (the manifest is ALREADY embedded as the trailing json
    fence — this is the chosen manifest-read source, see §1d).
- `updateTodo(project, leaf.id, { link: { blueprintId: <docId>, taskId: leaf.link?.taskId } })`
  → `src/services/todo-store.ts`. Sets `link.blueprintId` to the LATEST attempt's doc (the todo
  points at the most recent blueprint; prior attempts persist as their own docs, discoverable
  but not the primary link). Preserve any existing `taskId`.

```ts
persistBlueprint: async ({ project, leaf, attempt, blueprintMd }) => {
  const { createDocument } = await import('./document-manager'); // or the manager instance
  const { updateTodo } = await import('./todo-store');
  const id = await createDocument(
    `Leaf blueprint — ${leaf.id.slice(0,8)} attempt ${attempt}`,
    blueprintMd,
  );
  await updateTodo(project, leaf.id, {
    link: { blueprintId: id, ...(leaf.link?.taskId ? { taskId: leaf.link.taskId } : {}) },
  });
  return id;
},
```
(Confirm the real `document-manager` export form at wire time — it may be a manager singleton
method `documentManager.createDocument(...)` rather than a bare export; match the existing call
in `src/mcp/tools/documents.ts`.)

### 1d. Chosen manifest-read path: **(ii) embed the manifest in the persisted doc** — JUSTIFICATION
Three options were considered:
- (i) new field/endpoint carrying filesToCreate/Edit — extra surface, extra storage, redundant.
- (ii) **embed in the doc** — the manifest is ALREADY the trailing ```json fence of `blueprintMd`,
  and `parseSizeManifest` ALREADY extracts it from arbitrary text. So persisting the .md verbatim
  persists the manifest for free; the UI re-reads it via the SAME parser. ZERO new schema.
- (iii) extend `getLeafRun` to include the manifest — would need a new ledger column (manifest is
  not currently a ledger field) OR a side lookup inside ledger-stats, coupling the run-telemetry
  aggregator to blueprint docs. Heavier and mixes concerns.

**Pick (ii).** The doc is the durable single source; the todo's `link.blueprintId` is the pointer;
`parseSizeManifest` is the shared decoder. Minimal, additive, reuses P5 verbatim.

### 1e. Manifest-read ENDPOINT (small, additive)
Add ONE route in `src/routes/api.ts`, sibling to `GET /api/leaf-executor/run/:leafId`
(~line 2885) and `/stats` (~line 2887):

```
GET /api/leaf-executor/blueprint/:leafId?project=<p>
  → 200 { leafId, blueprintId } | { leafId, manifest: { filesToCreate[], filesToEdit[] } } | { leafId, ran:false }
```
Handler:
1. `getTodo(project, leafId)` → read `todo.link?.blueprintId` (todo-store). If absent → `{ leafId, ran:false }`.
2. `getDocument(blueprintId)` (document-manager) → its `content`.
3. `parseSizeManifest(content)` (imported from `../services/leaf-executor`) → null guard.
4. Return `{ leafId, blueprintId, manifest: { filesToCreate, filesToEdit } }` (only the two lists
   the card needs; keep the payload tight). Plain read, NO ws (b2fe36b1-compliant).

`project` query param: the strip already knows its project context indirectly; pass it from the UI
(TodoWorkerPanel has `project`). The new card will be given `project` + `leafId` like its siblings.
