# Blueprint ‚Äî [L3] Input plumbing: shared `cdp-input.ts` + `browser_input` WS

Leaf: `606eab8a-0860-4900-b4e6-5c9412fccb5f`
Sequenced after L2 (frame transport). Both touch the WS path; this leaf adds the
inbound (UI ‚Üí server ‚Üí CDP) half so the live panel can drive the page.

## Goal

1. **Pure refactor**: extract the inline `Input.dispatchMouseEvent` /
   `Input.dispatchKeyEvent` logic from `src/mcp/tools/browser.ts` into a new
   shared `src/services/cdp-input.ts`. The MCP browser_* tools call the shared
   helpers ‚Äî **zero behavioral change**.
2. **New inbound path**: handle a `browser_input` WS message (mouse / key /
   scroll), resolve the session's CDP target, translate canvas-relative coords ‚Üí
   page coords using the L2 frame metadata (`offsetTop`, `pageScaleFactor`,
   `deviceWidth`/`deviceHeight`), and dispatch via `cdp-input.ts`.

## Part A ‚Äî `src/services/cdp-input.ts` (NEW)

Factor the four inline CDP-Input call sites in `browser.ts` into typed helpers.
The helpers take an already-connected CDP `client` (the `any`-typed
chrome-remote-interface client used everywhere in this codebase) so they work for
BOTH the MCP tools (which already hold a `client` inside `withCDPSession`) and the
panel path (which connects its own client). Keep them `client`-first ‚Äî do NOT
have them open/close CDP connections; the caller owns the connection lifecycle
(matches `withCDPSession` ownership in `cdp-session.ts:128-165`).

Functions to export (mirror the exact current dispatch shapes ‚Äî see citations):

```ts

export async function pressKey(client: any, key: string): Promise<void>

// per-char keyDown/keyUp with text+key  ‚Äî from browserTypeText (browser.ts:247-248)
export async function typeText(client: any, text: string): Promise<void>

// dispatchMouseEvent({type:'mouseMoved', x, y})  ‚Äî from browserHover (browser.ts:177)
export async function mouseMove(client: any, x: number, y: number): Promise<void>

// mousePressed‚ÜímouseMoved‚ÜímouseReleased  ‚Äî from browserDrag (browser.ts:237-239)
export async function drag(client: any, sx: number, sy: number, tx: number, ty: number): Promise<void>

// NEW (no existing inline site): a discrete click at page coords for the panel.
// Implemented as mousePressed + mouseReleased (button:'left', clickCount:1) ‚Äî the
// same primitive drag() uses, just same-point. Used by the browser_input handler.
export async function click(client: any, x: number, y: number,
  button: 'left'|'middle'|'right' = 'left'): Promise<void>

// NEW: wheel/scroll for the panel ‚Äî Input.dispatchMouseEvent({type:'mouseWheel', x, y, deltaX, deltaY})

// NEW: a key event that also carries text/modifiers for the panel keydown path.
// type:'keyDown' then 'keyUp'; optional text + modifiers bitmask passed through to CDP.
export async function key(client: any, opts: {
  key: string; text?: string; code?: string; modifiers?: number;
  type?: 'keyDown'|'keyUp'|'char';
}): Promise<void>
```

The body of each helper is the **verbatim** `await client.Input.dispatch*` call(s)

## Part B ‚Äî `browser_input` inbound WS message

### Coordinate translation model

The screencast frame (L2) is a JPEG captured at up to
`MC_SCREENCAST_MAX_WIDTH/HEIGHT`, with metadata `{ offsetTop, pageScaleFactor,
deviceWidth, deviceHeight, timestamp }` broadcast per-frame in
`server.ts:211-222` and `handler.broadcastBrowserFrame` (handler.ts:453-461).

Page (CSS-pixel) coordinates that `Input.dispatchMouseEvent` expects are derived
from canvas-relative coords. To stay robust to the panel's display size, the UI
sends **normalized fractions** `xFrac,yFrac ‚àà [0,1]` of the rendered frame
(decision below). Server maps:

```
pageX = xFrac * deviceWidth
pageY = offsetTop + yFrac * deviceHeight
```

`deviceWidth`/`deviceHeight` are the CSS viewport dims; `offsetTop` accounts for
the visible-region scroll offset Chrome reports per frame. `pageScaleFactor` is
carried for completeness (pinch-zoom); under the default panel it is `1` and the
formula above is exact. If a future panel applies page scale, divide the mapped
coords by `pageScaleFactor` ‚Äî note this in a code comment but do not branch now.

The handler needs the **latest metadata per session**. Add a
`Map<string, FrameMeta>` in `server.ts` that the existing screencast sink
(server.ts:211) writes on every frame (`lastFrameMeta.set(session, {...})`),
cleared when the last viewer unsubscribes (server.ts:229-233) and on shutdown
(alongside `screencastUnsubscribers.clear()` at 606-608 / 622-624).

2. **server.ts** ‚Äî register the handler near the screencast bridge (after

   ```ts
   wsHandler.setOnBrowserInput(async (msg) => {
     const meta = lastFrameMeta.get(msg.session);
     try {
       await withCDPSession(msg.session, CDP_PORT, async (client) => {
         if (msg.action === 'key') {
           await cdpInput.key(client, { key: msg.key!, text: msg.text,
             code: msg.code, modifiers: msg.modifiers, type: msg.keyType });
           return;
         }
         // mouse/scroll need page coords from the latest frame meta
         if (!meta) return; // no frame yet ‚Äî drop (panel not streaming)
         const x = (msg.xFrac ?? 0) * meta.deviceWidth;
         const y = meta.offsetTop + (msg.yFrac ?? 0) * meta.deviceHeight;
         if (msg.action === 'scroll') {
           await cdpInput.scroll(client, x, y, msg.deltaX ?? 0, msg.deltaY ?? 0);
         } else if (msg.event === 'click') {
           await cdpInput.click(client, x, y, msg.button ?? 'left');
         } else if (msg.event === 'move') {
           await cdpInput.mouseMove(client, x, y);
         } else {
           // down/up ‚Äî dispatch a single press/release at the point
           await cdpInput.mousePress(client, x, y, msg.event!, msg.button ?? 'left');
         }
       });
     } catch (err) {
       console.error(`mermaid-collab: browser_input dispatch failed for ${msg.session} ‚Äî`, err);
     }
   });
   ```
   - Import: `import * as cdpInput from './services/cdp-input.js';` and reuse the
     existing `withCDPSession, CDP_PORT` imports already used elsewhere
     (`browser.ts` imports them from `cdp-session.js`; confirm/add the import in
     `server.ts`).
   - Add a small `mousePress(client, x, y, 'down'|'up', button)` helper to
     `cdp-input.ts` (single `dispatchMouseEvent` with
     `type: down?'mousePressed':'mouseReleased', clickCount:1`) so down/up drags
     from the panel are supported without the all-in-one `drag` helper.

   `withCDPSession` already resolves the session's CDP target from `tabRegistry`
   and throws a clear error if no tab is open (cdp-session.ts:135-139) ‚Äî reuse it
   rather than re-implementing target resolution. The panel and the screencast
   share the same `sessionName`, so the same registry entry is correct.

## Decisions (no human input needed)

- **Normalized fractions over raw canvas px**: the panel knows the rendered frame
  box; sending `[0,1]` fractions makes the server math independent of panel
  display size / DPR and avoids shipping canvas dims on every event. This is the
  one cross-cutting contract the L4 panel UI must honor ‚Äî call it out in the
  `browser_input` message JSDoc.
- **client-first helpers**: `cdp-input.ts` never opens CDP connections ‚Äî the
  caller (`withCDPSession` for both paths) owns the lifecycle. Keeps the refactor
  a pure extraction and avoids double-connecting.
- **Drop input when no frame meta yet**: mouse/scroll before the first frame is
  meaningless (nothing rendered); key events don't need coords so they pass
  through. No queueing.
