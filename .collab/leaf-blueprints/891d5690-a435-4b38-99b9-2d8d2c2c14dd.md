# Blueprint — [L2] Frame transport over WS + latency checkpoint

Leaf id: `891d5690-a435-4b38-99b9-2d8d2c2c14dd`

## Goal

Carry Chrome screencast frames produced by `ScreencastService` (L1) from the
server to browser clients over the existing WebSocket. Add a `browser_frame`
message variant + a `broadcastBrowserFrame()` fan-out on channel
`browser:<session>`, wire the ScreencastService sink to it, and — the real
deliverable — **measure** end-to-end frame latency/throughput under
realistic (VPN-like) conditions. Only escalate to a binary WS frame if the
JSON+base64 path is too slow. Record the measured numbers in the todo.

## Change shape

### 1. `src/websocket/handler.ts` — add the message variant

Add to the `WSMessage` union (insert before the `peer_registry` member on
line 130; keep the trailing `;` on the final member):

```ts
  | { type: 'browser_frame'; session: string; data: string; meta: {
      offsetTop: number; pageScaleFactor: number; deviceWidth: number;
      deviceHeight: number; timestamp?: number } }
```

### 2. `src/websocket/handler.ts` — add `broadcastBrowserFrame`

Add a method on `WebSocketHandler` (next to `broadcastBrowserTabUpdate`,
lines 430–432). It delegates to `broadcastToChannel` using the
`browser:<session>` channel so only clients watching that session receive frames:

```ts
  broadcastBrowserFrame(session: string, frame: {
    data: string;
    meta: { offsetTop: number; pageScaleFactor: number; deviceWidth: number;
            deviceHeight: number; timestamp?: number };
  }): void {
    this.broadcastToChannel(`browser:${session}`, {
      type: 'browser_frame', session, data: frame.data, meta: frame.meta,
    });
  }
```

Note: `broadcastToChannel` already no-ops cheaply when there are zero
subscribers on that channel (the loop matches nothing), so emitting frames with
no viewers is harmless — but see §3 for the lazy gating that avoids even
capturing in that case.

### 3. `src/server.ts` — wire the ScreencastService sink to the broadcaster

The clean wiring respects ScreencastService's lazy lifecycle: subscribe a
WS-broadcasting sink **only while a client is watching** `browser:<session>`.

**Option A (recommended): demand-driven bridge keyed on WS channel subscribe.**
Add a small hook so the handler tells the server when a `browser:<session>`
channel gains its first / loses its last subscriber, and the server
subscribes/unsubscribes the ScreencastService sink accordingly:

- In `handler.ts`, add an optional callback field
  `private onChannelSubscriptionChange: ((channel: string, count: number) => void) | null = null;`
  with a setter, and invoke it inside `handleMessage` after a channel
  subscribe (line ~172) and after a channel unsubscribe (line ~198), passing
  the channel string and the current number of connections holding that
  channel key.
- In `server.ts` after line 193, register the callback (guard on
  `screencastService` non-null — only armed in `streamed-panel` mode). When a
  `browser:<session>` channel transitions 0→1, call
  `screencastService.subscribe(session, sink)` and stash the returned
  unsubscribe fn in a `Map<string, () => void>`; when it goes 1→0, call the
  stashed unsubscribe fn and delete the map entry. The `sink` is:
  ```ts
  (frame) => wsHandler.broadcastBrowserFrame(frame.sessionName, {
    data: frame.data,
    meta: {
      offsetTop: frame.metadata.offsetTop,
      pageScaleFactor: frame.metadata.pageScaleFactor,
      deviceWidth: frame.metadata.deviceWidth,
      deviceHeight: frame.metadata.deviceHeight,
      timestamp: frame.metadata.timestamp,
    },
  });
  ```

**Option B (simpler, eager): always-on sink.** Rejected as default because
`subscribe` needs a concrete sessionName and would start CDP capture with no
viewers. Document if chosen.

Add the unsubscribe-map cleanup to the existing shutdown handlers near lines
566/580 (call each stashed unsubscribe, then `screencastService?.stop()`).

### 4. CHECKPOINT — latency/throughput measurement (the real deliverable)

Before any UI is built on this, measure the JSON+base64-over-WS path. Method:

1. Run the server in `streamed-panel` mode
   (`MC_BROWSER_TARGET=streamed-panel`), owned Chrome navigated to a
   non-trivial animated page.
2. Instrument timing without permanently shipping perf code into the hot path —
   a temporary `performance.now()` around the sink, or a throwaway measurement
   client. Capture:
   - **frame size** (bytes of the JSON string and of raw base64),
   - **end-to-end latency**: CDP `metadata.timestamp` (capture) →
     client `onmessage` receive time,
   - **throughput / achieved FPS** vs the configured `everyNthFrame`/quality,
   - **CPU/GC pressure** of `JSON.stringify` on large base64 strings.
3. Simulate VPN-like conditions: add latency + constrained bandwidth (macOS
   Network Link Conditioner, or `tc`/`comcast` on Linux, or Chrome DevTools
   client-side throttling). Record numbers for at least: localhost,
   ~50ms RTT / 5Mbps, ~150ms RTT / 2Mbps.

**Decision gate.** If base64-over-WS sustains acceptable interactive latency
(rule of thumb: < ~200ms added latency and stable FPS at target quality on the
worst simulated profile), keep the JSON path — simplest, reuses
`broadcastToChannel` verbatim. **If too slow** (base64 inflates payload ~33%
and `JSON.stringify` over hundreds of KB per frame is the suspected
bottleneck), switch to a **binary WS frame** before building UI:
- Send `ws.send(arrayBuffer)`: compact header (type tag + session id + the
  small meta struct) followed by **raw JPEG bytes** (decode base64 once on the
  server). Removes base64 inflation and per-frame `JSON.stringify`.
- Client distinguishes binary vs text (`event.data instanceof ArrayBuffer/Blob`).
- Keep `broadcastBrowserFrame`'s signature; change only its internal encoding
  and add a binary channel fan-out paralleling `broadcastToChannel`.

**Document the measured numbers in the todo on completion** (per-profile size,
latency, FPS, and the keep-JSON-vs-go-binary decision with justification).

## Risks / notes

- Frame fan-out can flood the WS with no backpressure handling. Option A's
  demand-driven gating ensures capture runs only while a viewer is attached —
  the main mitigation.
- `metadata.timestamp` is CDP monotonic seconds — fine for relative latency
  deltas if the client reads a comparable clock; otherwise measure
  server-sink→client-receive with a wall clock stamped at send time.
- The binary fallback is pre-scoped here but implement it only if the
  checkpoint says JSON is too slow — don't build both.
