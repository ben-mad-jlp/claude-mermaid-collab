### Exact implementation shape

```ts
import { WebSocketClient, getWebSocketClient } from './websocket.js';

// Cache for cross-origin per-server clients, keyed by server.id
const frameClientCache = new Map<string, WebSocketClient>();

/**
 * Build the WS URL for a given server.
 * Same-origin (or no server): mirrors getDefaultWebSocketURL() from websocket.ts.
 * Cross-origin: explicit host:port.
 */
export function frameWsUrl(server?: { host: string; port: number }): string {
  if (typeof window === 'undefined') {
    return 'ws://localhost:9002/ws';
  }
  const proto = window.location.protocol === 'https:' ? 'wss:' : 'ws:';
  if (!server || `${server.host}:${server.port}` === window.location.host) {
    return `${proto}//${window.location.host}/ws`;
  }
  return `${proto}//${server.host}:${server.port}/ws`;
}

/**
 * Return the WebSocketClient bound to the given server (C3: per-server, not global singleton).
 *
 * Same-origin path: returns the existing shared singleton (getWebSocketClient()) —
 * it IS that server's connection in the streamed-panel same-origin deployment.
 *
 * Cross-origin path: lazily creates and caches a new WebSocketClient per server.id,
 * calls connect() once, and reuses on subsequent calls.
 *
 * Known gap: cross-origin servers protected by an Authorization header cannot be
 * reached over a browser WS connection (no custom headers on WS handshake).
 * The shipping streamed-panel target is same-origin, so this is a deferred follow-up.
 */
export function getFrameClient(server?: { id?: string; host: string; port: number }): WebSocketClient {
  if (typeof window === 'undefined') {
    // Test / SSR: always return the singleton (no real WS anyway)
    return getWebSocketClient();
  }

  const isSameOrigin = !server || `${server.host}:${server.port}` === window.location.host;

  if (isSameOrigin) {
    return getWebSocketClient(); // existing singleton
  }

  // Cross-origin: need a dedicated client per server
  const key = server.id ?? `${server.host}:${server.port}`;
  if (!frameClientCache.has(key)) {
    const client = new WebSocketClient(frameWsUrl(server));
    frameClientCache.set(key, client);
    client.connect(); // fire-and-forget; connect() is idempotent
  }
  return frameClientCache.get(key)!;
}

/** For testing: clear the cross-origin client cache. */
export function resetFrameClientCache(): void {
  frameClientCache.clear();
}
```

### Design decisions

1. **Same-origin fast path** — returns the module-level singleton from `websocket.ts`. This is correct because the streamed-panel deployment is a plain Chrome tab served by the same server; there is only ever one WS connection needed.

2. **Cross-origin path** — `Map<string, WebSocketClient>` keyed by `server.id` (falling back to `host:port` if no `id`). `connect()` is called once at creation time. Do NOT call `disconnect()` on removal (clients are shared across panel instances).

3. **`connect()` idempotency concern** — the blueprint says "idempotent" but the actual `WebSocketClient.connect()` creates a new `WebSocket` each call if one isn't open (it does NOT guard against double-connect internally). The cross-origin path in `getFrameClient` calls `connect()` only once (at cache-miss time), so callers (`StreamedViewport`) should also `await client.connect()` before subscribing — but because the cross-origin client is already connecting/connected by the time `getFrameClient` returns, the await in the effect will resolve quickly. For the same-origin singleton, callers must also `await client.connect()` as they do today.

4. **No `disconnect()` on cache eviction** — the cache is module-scoped and lives for the app lifetime. This is intentional: `StreamedViewport` components share the same underlying client, so one unmounting should not close the connection used by another panel. The client is only ever closed by `resetWebSocketClient()` / `resetFrameClientCache()` in tests.

5. **SSR/test guard** — `typeof window === 'undefined'` check falls through to the singleton, matching `getDefaultWebSocketURL()`'s own SSR guard.
