# Blueprint ‚Äî P4a: Ledger aggregation: per-leaf/per-node stats API (backend)

Leaf id: `42e7d9d4-37df-4d2c-b010-e43ea812df3e`
Branch: `collab/epic/5bcd5b8d` (P1+P2+P3 built)

## 0. Purpose / scope

P4a is the **read-side backend data source** that two later leaves consume:
- the per-todo **run view** (live node progress for ONE leaf), and
- **P4b** (fleet/aggregate stats).

It aggregates the existing worker-ledger NODE rows (written by P2's `runLeaf` via
`recordNode`) into queryable shapes, exposed as **plain GET endpoints** the Bridge
reads. NO new WS event types (constraint **b2fe36b1**); the API derives entirely from
existing ledger reads + existing ws (constraint **00c8adb9**: reuse/compose ‚Äî this leaf
composes `queryLedger` and mirrors the `/api/worker-run` route block, it invents no new
storage or transport).

This leaf is **READ-side only**. The one writer-side gap (verdict/outcome not recorded)
is documented in ¬ß4 with the minimal additive change, but is **flagged, not required**,
for P4a's core value (the run-view order/attempts/budget/authMode all already exist).

## 1. What exists today (verified)

### worker-ledger.ts (`src/services/worker-ledger.ts`)
- SQLite via `bun:sqlite`, one DB file `~/.mermaid-collab/worker-ledger.db` (dir overridable
  by `MERMAID_SUPERVISOR_DIR`). Opened lazily by `openDb()`, cached in module `db`.
  `_closeLedgerDb()` resets the handle (test seam).
- Table `worker_ledger`, columns (post node-migration, all node cols nullable/additive):
  `id, project, todoId, epicId, session, phase, provider, model, source, inputTokens,
  outputTokens, costUsd, knownPrice(0/1), steps, parseError, nodeKind, nodesSpent,
  authMode, exitCode, durationMs, rateLimited(0/1 or NULL), leafId, ts`.
- **Read surface that EXISTS:**
  - `queryLedger(q: LedgerQuery): LedgerRow[]` ‚Äî filters `project`/`todoId`/`epicId`/`since`,
    `ORDER BY ts DESC, id DESC`, `LIMIT` (default 200, max 2000). Coerces `knownPrice`‚Üíbool,
    `rateLimited`‚Üíbool|null. **This is the primitive P4a builds on.**
  - `summarize(q): LedgerSummary` ‚Äî totals + `byPhase`/`byModel` rollups. Cost-focused; does
    NOT know about nodeKind/authMode/durationMs, so it is NOT sufficient for P4a's node stats.
  - `recordPhase`, `recordNode` (writers).
- **No per-leaf and no node-stat query helper exists.** `queryLedger`/`summarize` are the
  only reads. P4a ADDS the node-aware aggregation.

### How recordNode is called today (`src/services/leaf-executor.ts`, `runNode`, ~L204-231)
Per node, `runLeaf` writes:
`project, todoId(=leaf.id), session(=leafSessionKey), epicId, leafId(=leaf.id),
nodeKind(='blueprint'|'implement'|'review'), model(from NODE_PROFILE),
nodesSpent:1, authMode(=res.authMode), exitCode(=res.exitCode), durationMs(=res.durationMs),
rateLimited(=res.rateLimited), inputTokens/outputTokens/costUsd/steps(from res.usage),
parseError`. `phase` defaults to `'node'`, `provider` to `'claude'`, `source` to `'node'`,
`ts` = now. ‚Üí **leafId, epicId, nodeKind, authMode, exitCode, durationMs, rateLimited, ts,
model are ALL present per node.**

**PRESENT for aggregation:** ordered node list (by ts), per-node kind/model/authMode/
exitCode/durationMs/rateLimited/ts, nodesSpent (sum = budget usage vs 20), wall-clock
(max ts ‚àí min ts, or Œ£ durationMs), attempts (countable as # of `blueprint` rows = # of
fresh attempts, since one blueprint starts each attempt), rate-limit/cap-pause (rows where
`rateLimited`=1), authMode audit (group by authMode).

**MISSING for aggregation (see ¬ß4):**
- **review verdict** (PASS/FAIL) ‚Äî `parseVerdict(review.text)` is computed in the executor but
  NOT persisted; `review.text` is not stored. Only the review node's `exitCode`/`parseError`
  land. A PASS/FAIL is therefore NOT derivable from the ledger today.
- **final leaf outcome** (`accepted|rejected|blocked|paused`) ‚Äî the `LeafRunResult.outcome`
  is returned to the caller, never written to the ledger. NOT derivable.
- **explicit attempt number per node** ‚Äî inferable (count of `blueprint` rows, or ts-ordering)
  but not stored as a column.

### Route idiom to mirror (`src/routes/api.ts`)
- Single big `handleAPI(req, ‚Ä¶): Promise<Response>` dispatcher; each route is
  `if (path === '/api/...' && req.method === 'GET') { ... return Response.json(...); }`.
- Services imported lazily inside the block: `const { queryLedger } = await import('../services/worker-ledger');`.
- **Closest analog = `/api/worker-run` block, `src/routes/api.ts:2844-2869`** ‚Äî reads
  `summarize` + `queryLedger`, collapses rows in JS, returns a derived JSON object. P4a's
  `/run/:leafId` mirrors this almost exactly (swap todoId‚ÜíleafId, phase-collapse‚Üínode-list).
- `/api/worker-ledger` block at `src/routes/api.ts:2783-2804` shows the query-param plumbing
  (project/since/limit) to copy.

### Bridge fetch idiom (consumer shape to fit)
`ui/.../bridge/WorkerRunSummary.tsx:50` does `fetch('/api/worker-run?'+qs)` then renders.
`LaneCallout.tsx` fetches `/api/worker-run` as the durable fallback. P4a endpoints must
return **plain JSON via GET**, fetchable the same way (no ws). The run-view leaf will
`fetch('/api/leaf-executor/run/'+leafId)`; P4b will `fetch('/api/leaf-executor/stats?...')`.

## 2. Design ‚Äî aggregation queries

New file: **`src/services/ledger-stats.ts`** (keeps node-stat aggregation out of the
write-path module; imports `queryLedger` + `LedgerRow` type from `worker-ledger.ts`). All
functions are pure over rows returned by `queryLedger`, so they unit-test against a seeded
temp DB exactly like `worker-ledger.test.ts`.

### 2a. PER-LEAF ‚Äî `getLeafRun(leafId, opts?)` ‚Üí `LeafRunStats | null`
Source rows: `queryLedger({ limit: 2000 })` filtered in JS to `r.leafId === leafId` (there is
no `leafId` filter in `LedgerQuery`; add one OR filter post-query ‚Äî prefer adding `leafId?` to
`LedgerQuery` + the `where` clause in `queryLedger`, a 2-line additive change, so the index can
be used; **note `idx` on leafId does not exist ‚Äî acceptable at current volume, document it**).
Re-sort ascending by `ts, id` (chronological run order).

Returns:
```ts
interface LeafNodeStat {
  nodeKind: string | null;      // 'blueprint'|'implement'|'review'
  model: string;
  authMode: string | null;
  exitCode: number | null;
  durationMs: number | null;
  rateLimited: boolean | null;
  ts: number;
}
interface LeafRunStats {
  leafId: string;
  epicId: string | null;
  project: string;
  nodes: LeafNodeStat[];        // chronological
  attempts: number;             // = count of nodeKind==='blueprint' rows (each starts a fresh attempt); min 1 if any nodes
  nodesSpent: number;           // Œ£ nodesSpent (fallback: nodes.length)
  nodeBudget: number;           // constant 20 (import NODE_BUDGET from leaf-executor)
  budgetPct: number;            // nodesSpent / nodeBudget
  wallClockMs: number;          // last.ts ‚àí first.ts (or Œ£ durationMs if a single-instant run)
  rateLimitedCount: number;     // nodes where rateLimited===true
  authModes: Record<string, number>;  // count by authMode (the audit, also surfaced fleet-wide)
  // derivable-only-if-recorded (see ¬ß4); null/undefined until executor writes them:
  finalOutcome?: 'accepted'|'rejected'|'blocked'|'paused'|null;
  reviewVerdict?: 'pass'|'fail'|null;
}
```
`finalOutcome`/`reviewVerdict` are populated ONLY if ¬ß4's optional executor change lands;
until then they are `null`. **The run-view's core (ordered nodes, attempts, budget bar,
wall-clock, authMode) needs NONE of the missing fields** ‚Äî it ships on existing data.

### 2b. FLEET / AGGREGATE ‚Äî `getFleetStats(opts?)` ‚Üí `FleetStats`
`opts = { project?, sinceTs?, epicId? }`. Source: `queryLedger({ project, epicId, since:
sinceTs, limit: 2000 })`, keep only node rows (`r.nodeKind != null` OR `r.leafId != null`),
group by `leafId`.
```ts
interface FleetStats {
  leafCount: number;
  nodesPerLeafAvg: number;        // Œ£ nodesSpent / leafCount
  attemptRate: number;            // Œ£ attempts / leafCount  (‚âà how often the 2nd attempt fires)
  blockRate?: number;             // leaves with finalOutcome==='blocked' / leafCount  (needs ¬ß4; omit/null until then)
  capPauseCount: number;          // node rows with rateLimited===true
  capPauseMs: number;             // Œ£ durationMs of rate-limited rows (best-effort; may be 0)
  authModeAudit: Record<string, number>;  // count by authMode across ALL nodes ‚Äî MUST be {subscription: N}; any 'api'/'unknown'/null key > 0 is an ALARM
  authModeAlarm: boolean;         // true iff any non-'subscription' authMode count > 0  (the subscription-only invariant, made first-class)
  wallClock: { p50: number; p90: number; max: number };  // per-leaf wallClockMs distribution
}
```
`authModeAudit` + `authModeAlarm` are the **subscription-only invariant made visible**
(¬ß4 of prompt point 4): the executor's `assertSubscriptionAuth` is the gate; this stat is the
post-hoc proof, computed from the recorded `authMode` of every node.

## 3. Design ‚Äî HTTP API (mirror `/api/worker-run` block exactly)

Add two GET blocks to `handleAPI` in `src/routes/api.ts`, placed adjacent to the existing
`/api/worker-run` block (~L2869), same lazy-import + `Response.json` idiom:

1. **`GET /api/leaf-executor/run/:leafId`** ‚Üí `getLeafRun(leafId)` JSON (or
   `{ leafId, ran:false }` when null, mirroring `/api/worker-run`'s `ran:false`). Match the
   path with a regex like `/^\/api\/leaf-executor\/run\/([^/]+)$/` (the file already uses
   `path.match(/^\/api\/diagram\/[^/]+\/history$/)` ‚Äî same idiom). Extract leafId from the
   capture group.
2. **`GET /api/leaf-executor/stats?project=&sinceTs=&epicId=`** ‚Üí `getFleetStats(opts)` JSON.
   Query-param plumbing copied from the `/api/worker-ledger` block (`url.searchParams.get`,
   `Number(...)` for sinceTs).

Both are plain reads ‚Äî **no ws, no new event type** (b2fe36b1 satisfied).

## 4. Writer-side gap ‚Äî minimal additive change (FLAGGED, optional)

The run-view's **core** ships without this. But `reviewVerdict` and `finalOutcome` (and a
crisp `blockRate`) require the executor to RECORD what it already computes. This is a small
change that **belongs in P2's `recordNode` usage** (leaf-executor.ts), so flag it back:

- **Minimal additive option (preferred):** in `runLeaf`, after `parseVerdict(review.text)`,
  pass the verdict into that review node's `recordNode` call via a new nullable column
  `verdict TEXT` (additive ALTER, same idiom as the node cols). And at each `return`, write a
  terminal marker row (or reuse the last node row) carrying `outcome TEXT`. Both are
  idempotent additive columns + nullable, so existing callers/tests are unaffected.
- **Strictly P4a-scoppable fallback (no executor edit):** derive a *weak* verdict signal from
  the review node's `exitCode===0 && parseError==null` ‚Äî but this is NOT the true PASS/FAIL
  (a clean exit can still be a FAIL verdict). **Do NOT present this as the verdict.** Leave
  `reviewVerdict`/`finalOutcome` = `null` and let the consuming leaf show "‚Äî" until the
  executor change lands.

**Recommendation:** ship P4a read-side now with the fields nullable; open a follow-up note
that the executor should record `verdict` + `outcome` (one-line additive). Do not block P4a.

## 5. authMode audit (first-class)
Implemented in both `getLeafRun.authModes` and `getFleetStats.authModeAudit`/`authModeAlarm`
(¬ß2). The invariant: every node MUST be `authMode==='subscription'`. Any other key (`'api'`,
`'unknown'`, or `null`) with count > 0 sets `authModeAlarm=true`. P4b renders the alarm; P4a
just computes it from recorded data.

## 8. Risks / blockers
- **R1 (data gap, flagged):** `reviewVerdict`/`finalOutcome` are NOT in the ledger today ‚Äî
  they need a small additive write in `leaf-executor.ts` (P2's recordNode usage). The run-view
  consuming leaf will show "‚Äî" for verdict/outcome until that lands. **This pushes a small
  change back into P2.** Everything else (nodes/attempts/budget/wall-clock/authMode) is fully
  available now. Recommend: ship read-side with nullable fields, file the executor follow-up.
- **R2 (attempts heuristic):** `attempts` is derived as `#blueprint rows`. Valid for the P2
  floor (each attempt starts with exactly one blueprint node). If P5/P6 changes the node
  sequence, revisit. Document the assumption in `ledger-stats.ts`.
- **R3 (no leafId index):** querying by leafId is unindexed (rows scanned under the 2000 cap).
  Fine at current volume; note for later (`CREATE INDEX idx_ledger_leaf`).
- **R4:** `bun:sqlite` test gate ‚Äî `ledger-stats.test.ts` must run under `bun test`, NOT
  vitest (it touches the real ledger DB), same exclusion as `worker-ledger.test.ts`.
- No new WS events, no new transport (b2fe36b1, 00c8adb9 satisfied).
