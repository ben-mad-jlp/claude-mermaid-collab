# Blueprint — P3: Rate-cap handling + daemon-owned pause/resume + circuit breaker

Leaf: `077b346e-7a3d-47d7-ba1f-e86fd35324e5`
Branch: `collab/epic/5bcd5b8d` (P1 node-invoker + P2 leaf-executor already built)

---

## 0. Ownership rule (the spine of this leaf)

> **The leaf-executor NEVER backs off, sleeps, or retries on a rate cap. It detects
> `rateLimited`, yields a distinct `paused` outcome with the run-state needed to
> resume, and returns immediately. The Orchestrator daemon owns ALL pause/resume/
> backoff/circuit-breaking.**

Concretely:
- `node-invoker` already exposes `NodeResult.rateLimited` (heuristic — see §5).
- `leaf-executor.runLeaf` adds a `paused` outcome; it does not loop on it.
- `orchestrator-live` (the daemon tick) holds the backoff state and suppresses new
  headless launches while a cap window is open, then re-dispatches paused leaves.

This keeps the executor a pure, synchronous state machine (still unit-testable with
mocks, no timers) and concentrates every timing decision in the one always-on loop.

## 1. Executor change — yield `paused`, do not backoff

### 1a. `LeafRunResult` gains a paused outcome (leaf-executor.ts:80-86)

```
export interface LeafRunResult {
  outcome: 'accepted' | 'rejected' | 'blocked' | 'paused';   // + 'paused'
  attempts: number;
  nodesSpent: number;
  reason?: string;
  // NEW — present only when outcome==='paused'. The minimum to resume.
  paused?: {
    atNode: LeafNodeKind;        // the node kind that hit the cap
    attempt: number;             // 1-based attempt in flight when paused
    nodesSpent: number;          // budget already consumed (carried across resume)
    capReset?: number;           // epoch ms the cap is known to reset, if known (§1c)
  };
}
```

### 1b. `runNode` surfaces rateLimited up to the loop (leaf-executor.ts:184-211)

`runNode` already returns the `NodeResult`; the *loop body* must check
`res.rateLimited` after EACH of the three `runNode` calls and short-circuit. Add a
tiny helper inside `runLeaf` so the check is uniform:

```
const pausedResult = (kind: LeafNodeKind, res: NodeResult): LeafRunResult => ({
  outcome: 'paused',
  attempts: state.attempt,
  nodesSpent: state.nodesSpent,
  reason: 'rate-limited',
  paused: { atNode: kind, attempt: state.attempt, nodesSpent: state.nodesSpent, capReset: res.capReset },
});
```

Then after each `await runNode(...)`:

```
const bp = await runNode('blueprint', buildSpec('blueprint', cwd));
if (bp.rateLimited) return pausedResult('blueprint', bp);
if (!checkBudget()) return parkBlocked('node-budget-exhausted');
// ...same after implement and review...
```

Order matters: the rate-limit check comes **before** the budget check and before
`parseVerdict` — a capped node produced no usable work, so we must not interpret its
(empty/error) output as a FAIL verdict (that would burn an attempt). A paused node
also should **not** advance `state.attempt`; because we `return` immediately, the
`for`-loop's `state.attempt += 1` never re-runs, so the attempt is preserved as-is.

> **Budget accounting on pause:** `runNode` increments `nodesSpent` BEFORE invoking
> (existing line 185). A capped node thus still counts 1 against the budget. This is
> acceptable and fail-safe (a capped node did consume a spawn slot). The daemon
> carries `nodesSpent` forward on resume (§3) so the master budget still bounds total
> spawns across pause/resume cycles.

### 1c. `capReset` — surface a reset time IF the CLI gives one

`node-invoker.ts` is the only place that sees raw stdout/stderr/json. Add an
OPTIONAL `capReset?: number` (epoch ms) to `NodeResult` and best-effort-parse it:

- **KNOWN:** nothing in `claude --help` documents a reset timestamp. So this is a
  best-effort scrape, gated behind the same RUNTIME-CONFIRM TODO as `RATE_LIMIT_RE`.
- Implementation: a `CAP_RESET_RE` that looks for an ISO-8601 / unix-epoch / `retry
  after Ns` token in stdout+stderr, parsed to epoch ms. If none found → `undefined`.
- The executor passes `res.capReset` straight through into `paused.capReset`; the
  daemon uses it if present, else falls back to exponential backoff (§2).

Fail-safe: a wrong/garbage `capReset` only changes *how long* the daemon waits; it
never loses work. If parsing is at all risky, ship `capReset` as always-`undefined`
(pure backoff path) and leave `CAP_RESET_RE` behind the runtime-confirm TODO. **v1
recommendation: ship `capReset` parsing as a no-op stub returning `undefined`** —
pure exponential backoff is correct and has no dependency on an unconfirmed signal.

## 2. Daemon circuit breaker (orchestrator-live.ts)

### 2a. Where the breaker state lives — new module `headless-breaker.ts`

A tiny in-memory, process-wide module (NOT a todo status — todo-store has no
`paused` state, §0, and per §"todo-store supports" the statuses are
`backlog|planned|todo|ready|in_progress|blocked|done|dropped`). In-memory daemon
state is the right home: it is transient, recomputed each tick, and dies with the
process (a restart simply re-attempts, which is safe).

`src/services/headless-breaker.ts` (NEW):

```
// Process-wide breaker for headless (node-invoker) launches.
let openUntil = 0;          // epoch ms; launches suppressed while now < openUntil
let consecutiveTrips = 0;   // for exponential backoff when capReset unknown

const BASE_BACKOFF_MS = 60_000;       // 1 min
const MAX_BACKOFF_MS  = 30 * 60_000;  // 30 min ceiling
const MAX_TOTAL_WAIT_MS = 2 * 60 * 60_000; // 2h → exhaustion (§4)

export function tripBreaker(capReset?: number, now = Date.now()): void {
  consecutiveTrips += 1;
  if (capReset && capReset > now) {
    openUntil = Math.max(openUntil, capReset);
  } else {
    const backoff = Math.min(BASE_BACKOFF_MS * 2 ** (consecutiveTrips - 1), MAX_BACKOFF_MS);
    openUntil = Math.max(openUntil, now + backoff);
  }
}
export function breakerOpen(now = Date.now()): boolean { return now < openUntil; }
export function breakerOpenUntil(): number { return openUntil; }
export function resetBreaker(): void { openUntil = 0; consecutiveTrips = 0; }
// Test seam: inject a clock.
```

Per-process (not per-project) is correct for v1: the cap is per-claude.ai-account
(`assertSubscriptionAuth`), and all headless nodes share that one account. A
per-project gate would let project B keep hammering the same capped account.

### 2b. Where the breaker is CHECKED in the tick

The breaker gates **new headless launches only** (tmux/legacy workers and the
reconcile/triage passes are untouched — they are not node-invoker spawns).

The claim loop is in **`coordinator-daemon.ts` `runTick`, lines 141-151** — that is
the seam. But `runTick` doesn't know a todo is "headless" until `launchWorker`
reaches `isHeadlessLeaf` (coordinator-live.ts:1427). Cleanest, least-invasive
placement: gate inside the existing LEAF_EXECUTOR branch in `launchWorker`
(coordinator-live.ts:1427-1439), BEFORE `runLeaf`:

```
if (leafExecutorEnabled() && isHeadlessLeaf(todo, project)) {
  if (breakerOpen()) {
    // cap window still open → do not spawn. Release the claim so the todo
    // returns to `ready` and is re-claimable on a later tick (lease also backstops).
    try { await releaseClaim(project, todo.id); } catch { /* lease backstops */ }
    return false;   // not spawned; no escalation — this is a transient hold
  }
  try {
    const res = await runLeaf(...);
    if (res.outcome === 'paused') {
      tripBreaker(res.paused?.capReset);
      enqueuePausedLeaf(project, todo.id, res.paused);   // §3
      try { await releaseClaim(project, todo.id); } catch {}
      return false;   // not accepted; will resume when breaker closes
    }
    // ...existing accepted/blocked handling...
  } catch (e) { /* existing escalate path */ }
}
```

Rationale for gating in `launchWorker` not `runTick`: `runTick` is provider-agnostic
and shared by tmux lanes; `isHeadlessLeaf` lives in coordinator-live. Gating here
keeps the breaker scoped to exactly the headless path and reuses the existing
release-on-no-spawn convention (the loop already treats `launchWorker → false` as
"claimed but not spawned"; the lease/`ready` re-surfacing handles retry).

> **Note / mild invasiveness flag:** suppressed-but-claimed todos rely on the lease
> expiry + re-claim to retry. Because we `releaseClaim` immediately, the todo goes
> back to `ready` and WILL be re-claimed next tick — and immediately re-suppressed by
> `breakerOpen()`. That's a tight spin (claim→release each tick) but cheap (no spawn).
> Acceptable for v1; a cleaner future option is a claim-time `claimGuard` filter that
> holds headless leaves out of the claimable set while `breakerOpen()` (mirrors the
> existing probe-gate filter at coordinator-live.ts:1167). **Recommended v1: add that
> claimGuard filter too** — it's ~4 lines and avoids the claim/release spin:
> in the `claimGuard` composition, drop `isHeadlessLeaf` todos when `breakerOpen()`.

## 3. Resume

### 3a. Paused-leaf registry — `headless-breaker.ts` (same module)

```
interface PausedEntry { project: string; todoId: string; paused: LeafPaused; }
const pausedLeaves = new Map<string, PausedEntry>(); // key = `${project}:${todoId}`
export function enqueuePausedLeaf(project, todoId, paused) {...}
export function drainResumable(now = Date.now()): PausedEntry[] {
  if (breakerOpen(now)) return [];      // window still open → resume nothing
  const out = [...pausedLeaves.values()];
  pausedLeaves.clear();
  return out;
}
```

When the breaker closes (`!breakerOpen()`), the daemon re-surfaces paused leaves.
Because we `releaseClaim` on pause (§2b), the todo is already back to `ready` and the
normal claim loop re-dispatches it — so the registry is mostly a *bookkeeping +
exhaustion-tracking* structure (it records `firstPausedAt` for §4), not a separate
dispatch path. The actual re-dispatch is the ordinary tick re-claiming a `ready`
todo once `breakerOpen()` is false.

### 3b. Resume strategy — **fresh-within-budget** (CHOSEN)

Two options:
1. **Mid-node re-entry:** persist `{worktree, blueprint, nodesSpent, atNode}` and
   resume at exactly the node that capped.
2. **Fresh-from-blueprint within the same attempt budget** (CHOSEN for v1): re-run
   `runLeaf` from the top; the blueprint file (`.collab/leaf-blueprints/<id>.md`) is
   already persisted in the epic worktree from P2, so a re-run's blueprint node is
   cheap/idempotent (re-writes the same file), and implement/review re-execute.

**Chosen: fresh-from-blueprint, carrying `nodesSpent` forward.** Rationale:
- The P2 executor is **stateless across calls** — it rebuilds a FRESH worktree every
  attempt (`fresh:true`, leaf-executor.ts:249) and has no resume-at-node machinery.
  Mid-node re-entry would require persisting node-level checkpoints the architecture
  doesn't have → large, invasive change for v1.
- The blueprint is the only durable artifact and it survives in the worktree; a
  re-run is therefore mostly-idempotent and correct.
- **Tradeoff:** a paused implement/review node re-does the blueprint node (one wasted
  spawn). To stop pause/resume cycles from defeating the master budget, the daemon
  passes the carried `nodesSpent` into the resumed run via a new optional dep
  `startNodesSpent?: number` on `LeafExecutorDeps` (defaults 0); `runLeaf` seeds
  `state.nodesSpent` from it. Thus total spawns across all pause/resume cycles stay
  bounded by `NODE_BUDGET` (20) — eventually a thrashing leaf parks BLOCKED on
  budget exhaustion (the existing `parkBlocked('node-budget-exhausted')` path).

> Net: resume = the ordinary claim loop re-running `runLeaf` once `breakerOpen()` is
> false, with `startNodesSpent` carried from the paused entry so the budget is global.

## 4. Exhaustion → park BLOCKED + escalate

The cap may persist indefinitely (account fully out of quota for the billing window).
Bound the total wait:

- `headless-breaker` tracks `firstTrippedAt` (epoch ms of the first trip in the
  current open streak; reset on `resetBreaker`/successful spawn).
- In the daemon tick, before re-dispatch, check
  `now - firstTrippedAt > MAX_TOTAL_WAIT_MS` (2h). If exceeded for a paused leaf:
  - call the EXISTING escalation API the executor already uses — `createEscalation`
    (imported in coordinator-live.ts:11; also `deps.escalate` in leaf-executor) with
    `kind:'blocker'`, the leaf's todoId, and a questionText naming rate-cap exhaustion.
  - park the todo BLOCKED via the existing funnel: route a `rejected` through
    `handleWorkerComplete` (same mechanism `parkBlocked` uses, leaf-executor.ts:215),
    which sets status `blocked` + clears completion. Then drop it from the paused map.

Simplest placement: a new daemon hook `sweepExhaustedHeadless(project)` called once
per tick (alongside the existing `escalateExhausted`/`detectStalls` hooks in
coordinator-daemon.ts:122-129), iterating `pausedLeaves` for the project and parking
any past the wait ceiling. Reuse `createEscalation` directly (no new API).

## 5. Detection reality — FAIL-SAFE

**KNOWN:**
- `NodeResult.rateLimited` is set by `RATE_LIMIT_RE` (node-invoker.ts:77) matched
  against stdout + stderr + parsed json `subtype`.
- A node's `ok` is already forced `false` when `rateLimited` (line 339).

**UNKNOWN (runtime-unconfirmed, per the existing TODO at node-invoker.ts:69-76):**
- The real 429/usage-limit exit code, stderr string, and json subtype are
  undocumented. `RATE_LIMIT_RE` is a heuristic; `CAP_RESET_RE` (§1c) even more so.

**Fail-safe design (must hold):**
- **False positive** (we flag a non-capped node as rateLimited): the leaf pauses and
  the breaker opens. Cost = a delay. The leaf re-dispatches when the (spurious) window
  closes and the next node runs normally. **No work is lost.** Bounded by §4.
- **False negative** (a real cap not matched by the regex): the node simply fails
  `ok:false` with a parse/error result → falls into the normal review-FAIL / attempt
  machinery → eventually `parkBlocked`. **Normal failure handling, no corruption.**

Therefore neither failure mode silently loses work — the requirement is met. The
RUNTIME-CONFIRM path: when a real 429 is first observed, tighten `RATE_LIMIT_RE` and
populate `CAP_RESET_RE` from the observed format; until then the broad regex +
pure-backoff path is the safe default. Keep the TODO comment and reference this
section from it.

**Breaker (headless-breaker.test.ts):**
- `tripBreaker()` with no capReset → `breakerOpen()` true; advance injected clock past
  `BASE_BACKOFF_MS` → false. Successive trips → exponential, capped at `MAX_BACKOFF_MS`.
- `tripBreaker(capReset=now+5min)` → `breakerOpenUntil()===capReset`.
- `drainResumable` returns [] while open, returns + clears entries once closed.
- Exhaustion: `firstTrippedAt` + clock past `MAX_TOTAL_WAIT_MS` → `sweep` parks blocked.

**Daemon integration (coordinator/orchestrator test with TickDeps/CoordinatorDeps
spies):**
- Breaker open → `launchWorker` on a headless leaf returns false WITHOUT calling the
  invoker (spy invoker uncalled), and `releaseClaim` called. (Or: claimGuard filter
  drops the headless leaf → not even claimed.)
- Simulate reset (advance injected clock past `openUntil`) → next tick re-claims the
  `ready` leaf and `runLeaf` runs to `accepted` with a now-healthy fake invoker.
- Exhaustion: clock past ceiling → `createEscalation` spy called once, todo routed to
  blocked via the `complete('rejected')` spy.

No timers fire in tests: `setInterval` is never started; `runOrchestratorTick`/
`runTick`/`runLeaf` are called directly with injected `now`/clock.

## Risks / blockers for a clean implement

1. **Rate-limit signal unconfirmed (medium):** `RATE_LIMIT_RE` is heuristic and
   `capReset` is undocumented. Mitigated by the fail-safe design (§5) and the
   pure-backoff default — implementable today; sharpen on first observed 429.
2. **claim/release spin (low):** without the claimGuard hold filter, a suppressed
   headless leaf claim/releases every tick. Cheap (no spawn) but noisy; the
   recommended claimGuard filter removes it. Slightly more than the spec's "one gate"
   — flagging it as marginally more invasive than §2 assumes.
3. **Resume is fresh-not-mid-node (by design):** wastes the blueprint node per resume.
   Bounded by the carried `nodesSpent` budget; acceptable v1 tradeoff (§3b).
4. **Per-process breaker vs multi-account future (low):** correct now (single
   subscription account); would need per-account keying if multi-account headless ever
   lands. Out of scope for v1.
