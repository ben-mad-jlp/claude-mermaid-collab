|---|---|---|

| `typeText(client, text)` | `browserTypeText` 246–249 | `for (const char of text) { await client.Input.dispatchKeyEvent({ type: 'keyDown', text: char, key: char }); await client.Input.dispatchKeyEvent({ type: 'keyUp', text: char, key: char }); }` |

⚠️ **Param-identity is the acceptance bar.** Note:
- `moveMouse` for hover emits `{ type: 'mouseMoved', x, y }` with **no button** field.
- `dragMouse`'s middle move DOES include `button: 'left'` but **no clickCount**; press/release
  include `clickCount: 1`.
- `typeText` includes BOTH `text` and `key` set to the char, with no `code`.
- `pressKey` sends only `{ type, key }` — no `text`, no `code`.

## NEW capabilities (panel-only, no existing call site — define per blueprint §Change 1)

```ts
// full click = press+release at one point.
clickMouse(client, x, y, button='left', clickCount=1):
  Input.dispatchMouseEvent({ type:'mousePressed',  x, y, button, clickCount })
  Input.dispatchMouseEvent({ type:'mouseReleased', x, y, button, clickCount })

// wheel scroll at a point.
scrollMouse(client, x, y, deltaX, deltaY):
  Input.dispatchMouseEvent({ type:'mouseWheel', x, y, deltaX, deltaY })

// single discrete key event (panel sends keydown/keyup separately).
dispatchKey(client, type, opts):  // type: 'keyDown'|'keyUp'|'rawKeyDown'|'char'
  Input.dispatchKeyEvent({ type, ...opts })   // opts: { key?, text?, code?, windowsVirtualKeyCode?, modifiers? }

// single mouse press/release (mousedown/mouseup from panel).
dispatchMousePress(client, kind, x, y, button='left'):  // kind: 'mousedown'|'mouseup'
  const type = kind === 'mousedown' ? 'mousePressed' : 'mouseReleased'
  Input.dispatchMouseEvent({ type, x, y, button, clickCount: 1 })
```

`dispatchMousePress` is required by server.ts §3e (referenced as `cdpInput.dispatchMousePress`)
— the blueprint's helper-signature list in §Change 1 omits it but §3e mandates it. **Include it.**
Sensible defaults to mirror existing code: `button` defaults to `'left'`, `clickCount: 1`.

For `dispatchKey`, build the param object by spreading `opts` after `type` so undefined fields
are simply absent. Recommended: pass through only defined keys (or spread `opts` directly — CDP
tolerates extra undefined props, but cleaner to spread). Either is acceptable for tests since the
test asserts the recorded object; prefer `{ type, ...opts }`.

## Translation helper (exported for unit-testing)

```ts
export interface BrowserFrameMeta { offsetTop: number; pageScaleFactor: number; deviceWidth: number; deviceHeight: number }

export function canvasToPage(meta: BrowserFrameMeta, nx: number, ny: number): { x: number; y: number } {
  const x = (nx * meta.deviceWidth) / (meta.pageScaleFactor || 1);
  const y = meta.offsetTop + (ny * meta.deviceHeight) / (meta.pageScaleFactor || 1);
  return { x, y };
}
```
- `nx`/`ny` are normalized [0,1].
- `|| 1` guards a 0/undefined `pageScaleFactor`.
- **Out of scope (flag, don't add):** no horizontal scroll-offset term — would need an L2 meta
  extension (`scrollOffsetX/Y`). Add a code comment noting this so it isn't silently dropped.
