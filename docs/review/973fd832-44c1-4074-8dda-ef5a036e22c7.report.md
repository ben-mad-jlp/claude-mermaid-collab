**Finding 2 (real, evidence-backed) ŌĆö the once-per-epic base-gate cache never invalidates when the epic's base actually moves, so forward-integration can silently trust a stale verdict**
`ensureBaseGreen` (leaf-executor.ts:2050-2081) reads `getEpicBaseGate(epicId)` and returns the cached row **unconditionally** ŌĆö it never compares `cached.baseSha` to the current `epicBaseSha` (leaf-executor.ts:1911, computed fresh every run) before trusting the cache. Yet leaf-executor.ts:1889-1908 runs `wm.forwardIntegrateEpic(epicId, 'master')` on **every** leaf dispatch, which can genuinely change the epic branch's content by merging in new master commits. Two failure directions:
  - *Stale-red*: base gate cached `'error'`/`'fail'` at an early base; later forward-integration brings in the fix; every leaf stays wrongly parked forever (compounds Finding 1 ŌĆö no exposed way to clear it).
  - *Stale-green (the more dangerous direction)*: base gate cached `'pass'` at an early base; a later forward-integration merges a since-broken master into the epic branch; `ensureBaseGreen` still returns the old cached `'pass'` and no leaf ever re-checks. This is exactly the failure mode G2 exists to prevent ("A red base is the most important fact on a branch: EVERY leaf built on it inherits its brokenness") ŌĆö but it can happen invisibly post-forward-integration because the cache is keyed on `epicId` alone with no sha comparison, despite `baseSha` being stored in the row for exactly this purpose and never read back.
- File: `src/services/leaf-executor.ts:2054-2064` (cache-hit branch ignores `cached.baseSha`), `src/services/leaf-executor.ts:1889-1911` (forward-integration + baseSha computed fresh each run, but never reconciled against the cache)
