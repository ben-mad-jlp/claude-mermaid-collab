### 3. G2 — `final = mechanical AND llm` — **MET (in code)**

**Adversarial probe — can I construct a review-node output that causes an accept on a tree with a failing test?** **No, not via the lattice.** The review node's text is only consulted after `mech.status === 'pass'`, and its only reachable outputs there are `'pass'` (→ G3 grounding gate) or `'fail'`. ## Findings (4 session-todos filed)

**1 — `gate.test` uses the wrong runner for UI specs (material).** `.collab/project.json:8` declares `"test": "bun test {file}"` with no `testCwd`. With an empty prefix, `specFilesInChangeSet` returns *every* spec in the change-set, and `SPEC_FILE_RE` (`gate-runner.ts:358`) matches `ui/**/*.test.tsx`. This repo is dual-runner. I ran the real command:

```
bun test ui/src/components/ArtifactLink.test.tsx
→ ReferenceError: beforeEach is not defined
→ 0 pass, 0 fail, 1 error   (non-zero exit)
```

Any leaf touching a UI spec therefore gets `mech.status = 'fail'`, `gateFindingsText()`, and a fix node driven against a **phantom finding** — reproducing precisely the implement→review thrash this epic was built to eliminate. The pre-existing `changeSetTestCommand`/`changeSetTestCwd` keys (`project.json:4-5`) encode the bun/vitest split; the new `gate` block ignores it, and `LeafGateConfig` (`leaf-gate.ts:17-28`) can express only one command and one cwd, so the split is currently inexpressible.

**2 — a transient base-gate `'error'` is cached forever.** `leaf-executor.ts:2072-2079` calls `recordEpicBaseGate` unconditionally with `r.status`, so `'error'` (gate *could not run*: missing tool, OOM, signal-kill) persists exactly like a `'fail'`. The first leaf escalates and parks; every subsequent leaf reads the cache, gets `fresh:false`, and parks at `1494` with **no escalation at all**, until a human calls `clearEpicBaseGate`. This contradicts the adjacent comment at `2052` — "never cache a config error as a base fact" — which guards only the `misconfigured` *declaration* case, not a runtime `'error'`. An incident is not a base fact.

**3 — an unwired `deps.runGate` silently restores LLM-ratifies.** `leaf-executor.ts:1707-1708` falls back to `?? { status: 'pass', declared: false }`, after which `composeVerdict('pass', llm)` returns the LLM's verdict verbatim. The `absent`-declaration path was deliberately made loud by G4 (`warnGateAbstention` + a `gate-abstain` ledger row, `2026-2042`); the unwired-dep path does neither. Relatedly, `LeafGateResult.declared` is documented at `leaf-gate.ts:43-47` as "the ONLY way to get a `pass` without a command running" — yet the call site never reads it, so nothing downstream distinguishes "gate ran and passed" from "no gate ran."

**4 — duplicate `gate-abstain` ledger rows (cosmetic).** `runGate` is called once per review-loop iteration, and its `recordNode({nodeKind:'gate-abstain'})` at `2030-2042` lacks the module-level `Set` dedupe that guards `warnGateAbstention` (`892-900`). `nodesSpent:0`, so no budget impact.

*Not filed:* each of the seven commits appears **twice** with an identical subject and timestamp (`d4d50275`/`2e6ce5e0`, `85ee2011`/`6d3fe271`, …), suggesting a double-commit in the executor's commit path. Out of scope for this spec; noted for the record.

## Assessment

All four locked decisions are met **in the code under review** — the lattice is sound, the tri-state floor is real, G0 is correct and junk-free, and G3's validator is matched by a correspondingly updated prompt. I could not construct any review-node output that ratifies past a red gate.

The change-set nonetheless ships a configuration (`.collab/project.json:8`) that will fire the mechanical gate's `'fail'` arm on a false positive for any leaf touching a UI spec, driving a fix node against a phantom finding — the exact failure mode the epic exists to repair, reintroduced one layer up. That is a material gap in the change-set as delivered, and findings 2 and 3 are real holes in the "an incident is never silent" and "the LLM never ratifies" guarantees the epic asserts.
