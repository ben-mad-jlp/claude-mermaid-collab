## Findings

### Finding 1 — CRITICAL — `create_gate` MCP tool throws `MissingKindError` on its default (unparented) path
**File:** `src/services/todo-store.ts:1390-1404` (`createGate`), interacting with `todo-store.ts:809-834` (`resolveTodoParent`).

`createGate` builds its `createTodo` input from `CreateGateInput` (`todo-store.ts:1359-1373`), which has **no `kind` field** — unlike every other production caller (`session-todos.ts:356/426`, `api.ts:3658`, `todo-migration.ts:58`), which all pass `kind` explicitly. `resolveTodoParent` calls `isEpic(input)` (→ `kindOf(input)`) **before** any default is applied, and `kindOf` now throws `MissingKindError` on a missing `kind` (by design — "a missing kind is a bug at the producer, not a default", `todo-kind.ts:30-35`).

**Failure scenario:** call the `create_gate` MCP tool with the documented default — "Omit to leave the gate unparented" (`parentId` omitted, gate attaches via `dependsOn` instead). `resolveTodoParent`'s early-return (`if (input.parentId) return input.parentId`) doesn't fire, falls through to `isEpic(input)`, and throws. **Verified live:** `bun test src/services/__tests__/todo-store.test.ts -t createGate` → 6/6 fail, full stack trace through `resolveTodoParent`/`createTodo`. This is a real, currently-broken production capability (readiness gates), not a test-only artifact — `createGate` was simply missed when the "every producer states kind explicitly" rule was rolled out.

### Finding 2 — MAJOR — Backend test suite is broadly red on this branch; the breakage cross-contaminates unrelated tests
**Files:** `src/services/__tests__/todo-store.test.ts`, `todo-store-kind-stage-c.test.ts`, and (via pollution) most of `src/services/__tests__/*`.

Running `bun test src/services/__tests__/*.test.ts` on this branch: **296 of 1507 tests fail.** Root cause is the same as Finding 1's pattern, at test-fixture scale: dozens of `createTodo(project, {...})` calls across the suite were never updated to pass `kind`, so they throw `MissingKindError` at `resolveTodoParent`. In `todo-store.test.ts` alone that's 104/129 failing; `todo-store-kind-stage-c.test.ts` is 7/9 failing.

Worse: because a thrown `createTodo` mid-test skips its own cleanup, failures **leak between tests in the same file** — e.g. `mission is never rolled up / never epic-ready-to-land > kind-only: ...` (the direct regression test for spec item 8) fails when the full file runs, but **passes when run in isolation** (`bun test todo-kind-migration.test.ts -t "mission is never rolled up"` → 2/2 pass). Same pattern hit `container drop → cascade-drop undone descendants` (7 fails) and `completeTodo epic roll-up` (8 fails) — tests that directly cover this epic's own locked decisions (items 3-5, 8) are showing red in the branch's actual CI-equivalent run, even though the underlying implementation is correct when isolated.

Most strikingly, `todo-store-kind-stage-c.test.ts`'s own **"BOMB 1"** tests assert a contract the implementation now violates: *"a bracket-prefixed-LOOKING title with NO kind stores leaf, not the bracket role"* — the test expects `createTodo` with no `kind` to silently default to `'leaf'` (never throw, never regex the title). The actual code throws `MissingKindError` instead. This is either the test's documented intent being wrong, or `resolveTodoParent`/`kindOf`'s hard-throw-on-missing-kind policy being incompletely reconciled with `CreateTodoInput`'s optional `kind`. Either way it's a live contradiction inside the epic's own test suite, not a stale/unrelated failure.

**Why this matters:** the spec explicitly asks findings to be verified "by RUNNING code, not by reading it" for the catastrophic items — I did, and while the specific mechanisms (resolveEpicId, mission-rollup guards) are correct in isolation, the branch cannot currently claim a green test suite as evidence of that correctness, and one real capability (`create_gate`) is actively broken.
