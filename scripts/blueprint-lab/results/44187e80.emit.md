## Implementation Blueprint

### Problem

`_removeEpicInner` (`src/agent/worktree-manager.ts:2022-2041`) force-deletes the epic accumulation branch **unconditionally** after a land:

```ts
// The epic branch is now merged into master — delete it unconditionally (-D);
await this.runGit(this.opts.projectRoot, ['branch', '-D', branch], QUICK_TIMEOUT_MS)...
```

The comment *assumes* the branch is fully on master, but `branch -D` force-deletes even when it is not. If the land reported success while the branch still carries commits master does **not** have (partial land, gate/land bug, caller mis-gating — the exact 0949289b-family corruption class), those commits are destroyed with no ref left to recover them. There is already a durable, git-derived "is it fully on master?" signal — `epicAheadOfMaster()` (`worktree-manager.ts:973-984`, returns count of `master..<epicBranch>`; `0` ⇔ fully on master) — and a tip reader — `epicHeadSha()` (`worktree-manager.ts:1468-1482`). Neither is consulted before the destructive delete.

### Change shape

**1. `src/agent/worktree-manager.ts` — add a result type + make deletion conditional.**

Add an exported interface near the other epic types (e.g. just above `removeEpic`, ~line 2011):

```ts
export interface RemoveEpicResult {
  /** true iff the branch was safely deleted (was fully on master). */
  branchDeleted: boolean;
  /** commits the epic branch carried that master did NOT (0 ⇒ fully landed). */
  ahead: number;
  /** when NOT fully on master: the recovery branch the tip was renamed to. */
  retainedBranch?: string;
  /** when NOT fully on master: the recovered tip SHA (never lost). */
  retainedTip?: string;
}
```

Change `removeEpic` (`:2018`) and `_removeEpicInner` (`:2022`) return type from `Promise<void>` to `Promise<RemoveEpicResult>`. Rewrite the branch-deletion tail of `_removeEpicInner` (`:2034-2040`) as:

- Worktree `remove --force` + `prune` stay **unconditional** (unchanged `:2026-2033`) — the branch ref survives worktree removal independently, so this is always safe.
- Capture the tip **before** any branch mutation: `const retainedTip = await this.epicHeadSha(epicId);`
- `const ahead = await this.epicAheadOfMaster(epicId);` (reuses existing `resolveBase('master')` trunk logic).
- If `ahead === 0` (**fully on master**): `git branch -D <branch>` (existing call, swallowed for idempotency) → `return { branchDeleted: true, ahead: 0 };`
- Else (**not fully on master — flag + recover tip**): issue **no** `branch -D`. Rename the branch out of the active-epic namespace into a recovery namespace so it is flagged yet preserved: `git branch -m collab/epic/<id8> collab/gc-retained/<id8>` (mirror the exact `branch -m` form already in `renameEpicBranchToDropped` at `:2104-2108`; swallow failure so a missing/already-renamed branch stays idempotent). `return { branchDeleted: false, ahead, retainedTip: retainedTip ?? undefined, retainedBranch: 'collab/gc-retained/<id8>' };`

Renaming to `collab/gc-retained/*` keeps the tip reachable and pulls it out of `listUnlandedEpics()`'s `collab/epic/*` scan (`:1117-1138`), matching the "flag" intent (a dedicated, human-discoverable namespace) while never discarding commits.

**2. `src/services/coordinator-live.ts` — surface the flag at the one caller.**

At the `await wm.removeEpic(epicId, targetProject);` call (`:2302`), capture the result and, when `res.branchDeleted === false`, record a friction so the retained tip is visible (the `recordFrictionOnce` call already used in the adjacent catch at `:2304-2309`):

```ts
const gc = await wm.removeEpic(epicId, targetProject);
if (gc && !gc.branchDeleted) {
  await recordFrictionOnce(targetProject, {
    layer: 'operational',
    retryReason: 'landed-epic-not-fully-on-master',
    todoId: epicId,
    detail: `Land reported success but ${epicBranch} was ${gc.ahead} commit(s) ahead of master; branch retained as ${gc.retainedBranch} (tip ${gc.retainedTip}) instead of deleted.`,
  }).catch(() => {});
}
```

**3. `src/agent/__tests__/worktree-land.test.ts` — extend coverage.**

Mirror the existing `removeEpic deletes the epic branch + worktree (idempotent)` test (`:188-198`). Keep it (asserts the fully-on-master path still deletes and now returns `branchDeleted:true`), and add a test for the retain path: build an epic, land it, then add a **new commit to the epic branch after landing** (so `epicAheadOfMaster > 0`), call `removeEpic`, and assert (a) the original `collab/epic/<id8>` ref is gone, (b) `collab/gc-retained/<id8>` exists and points at the pre-delete tip, (c) the returned `branchDeleted === false` and `retainedTip` equals that SHA.

### Acceptance criteria (positive, citable)

1. `RemoveEpicResult` interface is exported in `worktree-manager.ts` and `removeEpic`/`_removeEpicInner` return `Promise<RemoveEpicResult>`.
2. `_removeEpicInner` calls `this.epicAheadOfMaster(epicId)` and only issues `git branch -D` when the result is `0`.
3. On the not-fully-on-master path, `_removeEpicInner` issues `git branch -m` into `collab/gc-retained/<id8>` and returns `retainedTip`/`retainedBranch` populated.
4. `coordinator-live.ts` inspects the `removeEpic` result and records a `landed-epic-not-fully-on-master` friction when `branchDeleted === false`.
5. A new named test in `worktree-land.test.ts` asserts the epic branch is retained-and-renamed (tip recovered) when ahead of master.

### Reuse notes
- `epicAheadOfMaster` (`:973`) and `epicHeadSha` (`:1468`) are existing, never-throw, git-derived helpers — do not reimplement the trunk resolution or SHA read.
- The `branch -m` form is copied verbatim from `renameEpicBranchToDropped` (`:2100-2110`); consider whether that method's `collab/dropped/*` namespace should be reused instead of a new `collab/gc-retained/*` — kept distinct here because the semantics differ (a *dropped* epic vs. a *land claimed done but wasn't* corruption signal), which the friction reason relies on.
