## Blueprint: `approveMission` store action + Approve button in the mission detail view

### Context (what already exists)
- The REST endpoint is already live: `POST /api/supervisor/missions/approve` taking `{ project, todoId }`, delegating to `handleMissionTool('approve_mission', body)` — `src/routes/supervisor-routes.ts:299-309`.
- The `'unapproved'` mission status is already in the UI vocabulary: `MissionStatus` union, `STATUS_STYLE`/`STATUS_LABEL`/`statusTooltip` all cover `unapproved` — `ui/src/components/supervisor/bridge/rail/missionShared.tsx:43,54,66`.
- The store holds a family of mission-authoring actions that all follow one shape: `invoke(...)` → guard `!res?.ok` → re-fetch via `get().fetchMissions(...)` — `ui/src/stores/supervisorStore.ts:971-993`.
- `MissionDetail` is the "mission detail view" (rendered by `MissionDetailPanel`). It pulls actions off the store via `useSupervisorStore((s) => s.xxx)` and dispatches them through a local `run()` wrapper that pipes the fresh list into `onChanged` — `missionShared.tsx:538-572`.
- `view.status` is the derived `MissionStatus` for the selected mission (`missionView`, `missionShared.tsx:740`).

This leaf wires the *missing* client half: a store action calling the existing route, and an Approve button in `MissionDetail` shown only while the mission is `unapproved`.

### Change 1 — store action `approveMission`

**File: `ui/src/stores/supervisorStore.ts`**

(a) Interface declaration — add immediately after the `activateMission` signature at line 583, mirroring its shape:
```ts
  approveMission: (serverId: string, project: string, todoId: string) => Promise<MissionSummary[]>;
```

(b) Implementation — add immediately after the `activateMission` impl (after line 975), mirroring it exactly but hitting the approve route:
```ts
  approveMission: async (serverId, project, todoId) => {
    const res = await invoke(serverId, '/api/supervisor/missions/approve', 'POST', { project, todoId });
    if (!res?.ok) return [];
    return get().fetchMissions(serverId, project);
  },
```
Note the route contract uses the key `todoId` (not `missionId`) — matches `supervisor-routes.ts:302` which requires `body.todoId`.
