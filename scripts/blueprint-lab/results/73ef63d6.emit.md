## Blueprint: Paginate `GET /api/supervisor/missions` + bounded single-scan fact collection

### Problem (grounded in current code)

`GET /api/supervisor/missions` (`src/routes/supervisor-routes.ts:235-245`) calls `listMissions(project, { session })` unconditionally and returns **every** mission with a **fully-computed rollup**. Per mission, `listMissions` (`src/services/mission-store.ts:987-1030`) does 3 full `listTodos` scans:

- `getMission` (`mission-store.ts:270-282`) → `collectMissionStatusFacts` → `listTodos` (line 865)
- `getMissionRollup` (`mission-store.ts:1093-1114`) → `listTodos` (line 1096)
- `getMissionRollup` → `collectMissionStatusFacts` → `listTodos` (line 865 again)

plus one root-enumeration scan at `mission-store.ts:991`. That's `1 + 3·N` synchronous `.all()` scans per request (the exact cost the `mission-loop.ts:32-44` comment documents), and **N is unbounded per request** — nothing caps how many mission summaries (with facts) are built.

The hot `listTodos` query is already protected from archive growth by the partial index `idx_todos_hot ON todos(status) WHERE archivedAt IS NULL` (`todo-store.ts:645`), so the 20k synthetic archive is the *proof fixture*: it demonstrates archived rows never leak into the response and never inflate the scan count.

### Change shape

**Two fixes:** (1) paginate the route so facts are computed for at most `limit` missions per request; (2) thread one shared hot-todos read through the per-mission builders so a whole page costs exactly **one** `listTodos` scan instead of `1 + 3·N`.

---

#### 1. `src/services/todo-store.ts` — test seam: count `listTodos` scans

Add a module-level counter incremented at the top of `listTodos` (`todo-store.ts:1365`), with exported reset/read seams (mirrors existing `_reset*`/`_closeProject` seams):

```ts
let _listTodosScans = 0;
export function _getListTodosScanCount(): number { return _listTodosScans; }
export function _resetListTodosScanCount(): void { _listTodosScans = 0; }
```
Increment `_listTodosScans++;` as the first statement of `listTodos`.

#### 2. `src/services/mission-store.ts` — thread a shared scan + paginate

**(a) Optional pre-read param on the three builders** (default preserves current behavior):

- `collectMissionStatusFacts(project, m, allTodosPre?: Todo[])` (line 862): `const allTodos = allTodosPre ?? listTodos(project, { includeCompleted: true });`
- `getMissionRollup(project, todoId, allTodosPre?: Todo[])` (line 1093): use `allTodosPre ?? listTodos(...)` for the epics filter (line 1096) and pass it into `collectMissionStatusFacts` (line 1102).
- `getMission(project, todoId, allTodosPre?: Todo[])` (line 270): pass `allTodosPre` into `collectMissionStatusFacts` at line 281.

**(b) Cheap raw reader** `getMissionRow(project, todoId): MissionRow | undefined` — same body as `getMission` **without** the `deriveMissionStatus(collectMissionStatusFacts(...))` scan (returns `rowToMission(row)` directly). Used only for page-membership decisions (archived filter) so non-page missions never pay for fact collection.

**(c) `listMissionsPage`** — new exported paginated reader:

```ts
export interface MissionSummaryPage { items: MissionSummary[]; nextCursor: string | null; }
export function listMissionsPage(
  project: string,
  opts: { session?: string; includeArchived?: boolean; onlyArchived?: boolean; limit?: number; cursor?: string | null } = {},
): MissionSummaryPage
```
Algorithm:
1. `const all = listTodos(project, { includeCompleted: true });` (the single scan)
2. `roots = all.filter(t => t.parentId == null && t.status !== 'dropped' && isMission(t))` (ord-ordered, as `listTodos` orders by `ord ASC`).
3. `pruneOrphanMissions(project, new Set(roots.map(t => t.id)))` (unchanged, line 998).
4. Build ordered `candidates`: for each root read `getMissionRow` (skip if none); apply the same archived filter as lines 1003-1004 on the raw row; apply the session filter (lines 1013-1015) using node fields. **No facts computed here.**
5. Clamp `limit = Math.max(1, Math.min(opts.limit ?? 50, 500))` (same clamp shape as `listArchivedTodos`, `todo-store.ts:1462`).
6. If `opts.cursor`, drop candidates up to and including the one whose `node.id === cursor`.
7. Take `limit` candidates → `page`; `nextCursor = (remaining > 0) ? page[last].id : null`.
8. For each page candidate build the full `MissionSummary` **passing `all`** into `getMission(project, id, all)` (keeps the terminal-active heal at lines 1009-1012) and `getMissionRollup(project, id, all)`; `criteria` via `listCriteria`; `epics` filtered from `all` (lines 1016-1018).

**(d) Refactor `listMissions`** (line 987) to also thread `all` for its internal callers — pass `all` into the `getMission` call (line 1001) and `getMissionRollup` call (line 1024). Return type unchanged (`MissionSummary[]`), so `assertMissionCreationAllowed`, `resolveActiveMissionId`, and `mission-loop.ts` are untouched, but each `listMissions` call now costs one scan instead of `1 + 3·N`.

#### 3. `src/routes/supervisor-routes.ts` — paginate the GET route

At `supervisor-routes.ts:235-245`, parse `limit`/`cursor` and delegate to `listMissionsPage`, preserving the `missions` response key (the UI reads `res.body.missions` at `ui/src/stores/supervisorStore.ts:969` and ignores unknown keys):

```ts
const limitParam = url.searchParams.get('limit');
const limit = limitParam ? Number(limitParam) : undefined;
const cursor = url.searchParams.get('cursor') ?? undefined;
const { listMissionsPage } = await import('../services/mission-store.ts');
const page = listMissionsPage(project, { session, limit, cursor });
return Response.json({ missions: page.items, nextCursor: page.nextCursor });
```

- `test('paginates GET /api/supervisor/missions to limit, exposes nextCursor')`: `handleSupervisorRoutes` on `?project=…&limit=2` → 200, `body.missions.length === 2`, `body.nextCursor` truthy; second call with `?cursor=<nextCursor>` → 1 item, `nextCursor === null`.

- `test('20k archived rows never leak into the missions page')`: every `items[].mission.archivedAt` is null and `items.length === min(3, limit)`.

### Acceptance criteria (positive, citable)

1. `src/services/mission-store.ts` exports `listMissionsPage` returning `{ items, nextCursor }` with a clamped `limit` and keyset-style `cursor`.
2. `src/services/mission-store.ts` `collectMissionStatusFacts`, `getMissionRollup`, and `getMission` each accept an optional `allTodosPre` and use it instead of calling `listTodos`.
3. `src/services/mission-store.ts` exports `getMissionRow` (raw read, no fact scan) used for page-membership filtering.
4. `src/routes/supervisor-routes.ts:235` GET handler calls `listMissionsPage` and returns `{ missions, nextCursor }`.
5. `src/services/todo-store.ts` exports `_getListTodosScanCount`/`_resetListTodosScanCount` and increments the counter in `listTodos`.
6. `src/services/__tests__/missions-list-bounds.test.ts` proves: pagination to `limit` with `nextCursor`, exactly one `listTodos` scan per `listMissionsPage` over a 20k archive, and no archived leakage.
