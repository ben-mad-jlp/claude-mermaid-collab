## Blueprint: Pin-as-conductor-target control in the mission rail + store action

### Context (what already exists)

The backend round-trip is **already shipped** (commit `15f01695`): `GET /api/supervisor/conductor` returns `{ project, enabled, targetMissionId }` and `POST /api/supervisor/conductor` accepts `{ project, enabled?, targetMissionId? }` and persists via `setConductorTargetMission` (`src/routes/supervisor-routes.ts:896-938`). This leaf is **UI-only**: expose the target in the client conductor store and add a rail control to set/clear it.

The conductor's client-side state lives entirely in the hook `ui/src/components/supervisor/bridge/useConductorEnabled.ts` (state + `setEnabled` action), **not** in `supervisorStore.ts`. That hook already hits the same endpoint that returns `targetMissionId`. So the correct, non-duplicative home for the "store action" is that hook — it is the conductor store. Adding a second setter in `supervisorStore.ts` would split one endpoint's state across two owners. The two current consumers (`ConductorStatusBadge.tsx:9`, `ProjectSettingsModal.tsx:216`) both destructure only the fields they use, so widening the result shape is safe.

The "mission rail" surface is `MissionDetail` in `ui/src/components/supervisor/bridge/rail/missionShared.tsx`, whose **Controls cluster** (lines 606-630) already holds Approve / Activate / Drop / Edit / Delete `MiniButton`s and already receives `project` + `serverId`.

### Change 1 — Conductor store action (the "store action")

**File:** `ui/src/components/supervisor/bridge/useConductorEnabled.ts`

```ts
  busy: boolean;
  setEnabled: (next: boolean) => Promise<void>;

}
```

**1b. Add target state + fetch it** in `useConductorEnabled` (line 23-37):
- Add `const [targetMissionId, setTargetState] = useState<string | null>(null);`
- Inside `fetchEnabled`, after the `enabled` read, add:
  `if (!cancelled) setTargetState(typeof data.targetMissionId === 'string' ? data.targetMissionId : null);`
  (The same GET already returns `targetMissionId`; no second request.)

**1c. Add the `setTargetMission` action**, mirroring `setEnabled` (line 39-48) exactly — optimistic set, POST `{ project, targetMissionId: next }`, reconcile from the response:

```ts
const setTargetMission = useCallback(async (next: string | null) => {
  if (!project) return;
  const prev = targetMissionId;
  setTargetState(next);                        // optimistic
  setBusy(true);
  const data = await apiPost('/api/supervisor/conductor', { project, targetMissionId: next });
  setTargetState(typeof data?.targetMissionId === 'string' ? data.targetMissionId : (data && 'targetMissionId' in data ? null : prev));
  setBusy(false);
}, [project, targetMissionId]);
```

**1d.** Return the two new members from the hook (line 50): `return { enabled, targetMissionId, busy, setEnabled, setTargetMission };`
