## Blueprint: Round-trip `targetMissionId` through GET/POST `/api/supervisor/conductor`

### Context (what already exists)

The store layer is **already built and tested** — `getConductorTargetMission(project)` / `setConductorTargetMission(project, missionId|null)` live at `src/services/supervisor-store.ts:483-493`, backed by the `watched_project.conductorTargetMissionId` column, and covered by `src/services/__tests__/supervisor-store-conductor-target.test.ts`. The conductor pass already consumes the pin (`src/services/conductor-pass.ts:96`).

The gap is purely the **HTTP surface**: the `GET`/`POST /api/supervisor/conductor` handlers in `src/routes/supervisor-routes.ts:894-912` only read/write `enabled` — they never expose or accept `targetMissionId`. This leaf wires the pin through those two handlers so a client can read and set it.

### Change 1 — import the two store setters

`src/routes/supervisor-routes.ts`, import block ending at line 33 (currently pulls `getConductorEnabled`, `setConductorEnabled` at lines 31-32). Add alongside them:

```ts
  getConductorEnabled,
  setConductorEnabled,
  getConductorTargetMission,
  setConductorTargetMission,
```

### Change 2 — GET returns `targetMissionId`

`src/routes/supervisor-routes.ts:895-899`. Extend the GET response object to also carry the pin:

```ts
  if (url.pathname === '/api/supervisor/conductor' && req.method === 'GET') {
    const project = url.searchParams.get('project');
    if (!project) return jsonError('project is required', 400);
    return Response.json({
      project,
      enabled: getConductorEnabled(project),
      targetMissionId: getConductorTargetMission(project),
    });
  }
```

`getConductorTargetMission` returns `string | null`, so the field is always present (null when unpinned).

### Change 3 — POST accepts + persists an optional `targetMissionId`

`src/routes/supervisor-routes.ts:902-912`. Keep `enabled` required (backward-compatible with the existing UI toggle in `ui/src/components/supervisor/bridge/useConductorEnabled.ts:44`). Accept `targetMissionId` **only when the key is present** in the body, so existing callers that send just `{ project, enabled }` are unaffected. Accept `string` (pin) or `null` (clear); reject any other type. Echo the persisted pin back:

```ts
  if (url.pathname === '/api/supervisor/conductor' && req.method === 'POST') {
    try {
      const { project, enabled, targetMissionId } = (await req.json()) as {
        project?: string; enabled?: boolean; targetMissionId?: string | null;
      };
      if (!project) return jsonError('project is required', 400);
      if (typeof enabled !== 'boolean') return jsonError('enabled must be a boolean', 400);
      if (targetMissionId !== undefined && targetMissionId !== null && typeof targetMissionId !== 'string') {
        return jsonError('targetMissionId must be a string or null', 400);
      }
      setConductorEnabled(project, enabled);
      if (targetMissionId !== undefined) setConductorTargetMission(project, targetMissionId);
      return Response.json({
        ok: true,
        project,
        enabled: getConductorEnabled(project),
        targetMissionId: getConductorTargetMission(project),
      });
    } catch (err) {
      return jsonError(err instanceof Error ? err.message : 'Unknown error', 500);
    }
  }
```

Note: like every other per-project setter here, `setConductorTargetMission` is UPDATE-only — it persists only for an already-watched project (no auto-watch). The route does not validate that the mission exists; `conductor-pass.ts:96-118` already resolves/clears a stale pin at drive time. Both are intentional and stay as-is.

### Change 4 — new route round-trip test

Create `src/routes/__tests__/supervisor-conductor.test.ts`, modeled on `src/routes/__tests__/supervisor-injection-flags.test.ts` (calls `handleSupervisorRoutes` directly, `addWatchedProject` first because the setters are UPDATE-only). Cover:

- `GET without project → 400`
- `GET returns targetMissionId (null when unpinned)` — asserts the key is present and `null` for a fresh watched project.
- `POST persists targetMissionId and echoes it` — `addWatchedProject`, POST `{ project, enabled: true, targetMissionId: 'mission-abc' }`, assert echo `targetMissionId === 'mission-abc'`, then `GET` reads it back as `'mission-abc'`.
- `POST with targetMissionId: null clears the pin` — after pinning, POST `{ project, enabled: true, targetMissionId: null }`, assert read-back is `null`.
- `POST omitting targetMissionId leaves the pin untouched` — pin, then POST `{ project, enabled: false }`, assert GET still returns the prior pin.
- `POST with a non-string, non-null targetMissionId → 400` — e.g. `targetMissionId: 42`.

Use distinct project paths per test (e.g. `/tmp/conductor-rt-a`, `-b`…) so cases don't cross-contaminate shared SQLite state.

### Acceptance criteria (positive, citable)

1. `src/routes/supervisor-routes.ts` imports `getConductorTargetMission` and `setConductorTargetMission` from `../services/supervisor-store.ts` (import block ~lines 31-33).
2. The GET `/api/supervisor/conductor` handler (~line 898) returns a JSON body containing a `targetMissionId` field sourced from `getConductorTargetMission(project)`.
3. The POST `/api/supervisor/conductor` handler (~line 902) destructures `targetMissionId` from the body, calls `setConductorTargetMission(project, targetMissionId)` when the key is present, and includes `targetMissionId: getConductorTargetMission(project)` in its response.
4. The POST handler returns a 400 when `targetMissionId` is present and is neither a string nor null.
5. `src/routes/__tests__/supervisor-conductor.test.ts` exists with a test asserting a POSTed `targetMissionId` is read back by a subsequent GET, and a test asserting `targetMissionId: null` clears the pin.
