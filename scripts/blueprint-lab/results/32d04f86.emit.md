## Blueprint — Conductor runs to LAND a build-green epic (don't building-wait past its land card)

### Problem (grounded)
In `src/services/conductor-pass.ts:104-152`, `runConductorPass` derives an actionability signal from the mission's per-criterion actions only:
- `conductor-pass.ts:124` — `hasGap = actions.some(discover|verify)`
- `conductor-pass.ts:126` — `if (!hasGap && status === 'building') return { ran:false, reason:'building-wait' }`

When the daemon has built an epic's leaves but the epic is **build-green + unlanded**, `mission-store.ts:648` derives that criterion's action as `building` (serving epic still `open` with live motion), so `hasGap` is false and status is `building` → the pass returns `building-wait` and never spawns the node that would call `land_epic`. Meanwhile the reconcile pass has already surfaced an OPEN `epic-ready-to-land` escalation (`reconcile-pass.ts:99`, `surfaceEpicLand`) — the exact signal the conductor should act on. The `land_epic` ownership gate already restricts the node to its own mission's epics, so making the pass *actionable* on an open land card is safe.

The debounce fingerprint (`conductor-pass.ts:49-52`) keys only on `status | per-criterion actions`. A freshly-surfaced land card changes neither, so even after we make the pass actionable, an unchanged fingerprint would debounce it away. The fingerprint must also include the open land-card count.

2. **Widen `conductorFingerprint`** (`conductor-pass.ts:49-52`) to fold in the open land-card count, with a defaulted param so the existing 2-arg pure call sites/tests stay valid:
   ```ts
   export function conductorFingerprint(status: string, actions: ConductorActionRow[], landCardCount = 0): string {
     const parts = actions.map((a) => `${a.id}:${a.action}`).sort();
     return `${status}|${parts.join(',')}|land:${landCardCount}`;
   }
   ```
   Update the doc comment above it to note that a newly-surfaced land card also moves the fingerprint.

3. **Compute the open land-card count and gate `building-wait` on it** inside `runConductorPass`, between the `actions`/`hasGap` block (`conductor-pass.ts:123-126`) and the fingerprint (`conductor-pass.ts:128`):
   ```ts
   const actions = listCriteriaWithActions(project, missionId).map((a) => ({ action: a.action, id: a.id }));
   const hasGap = actions.some((a) => a.action === 'discover' || a.action === 'verify');
   // An OPEN 'epic-ready-to-land' card for THIS project means a build-green epic is stranded
   // unlanded — actionable even when every criterion still reads 'building'. land_epic's
   // ownership gate ensures the node only lands its OWN mission's epics.
   const openLandCards = listEscalations('open').filter(
     (e) => e.project === project && e.kind === 'epic-ready-to-land',
   ).length;
   // 'building' with no discover/verify gap AND no open land card = the daemon is working;
   // nothing to direct — wait.
   if (!hasGap && openLandCards === 0 && status === 'building') {
     return { ran: false, reason: 'building-wait', missionId };
   }

   const fp = conductorFingerprint(status, actions, openLandCards);
   ```
   (The `stampConductorRun(project, missionId, fp)` at `conductor-pass.ts:150` already persists this widened fingerprint — no change there.)

Note: `listEscalations('open')` returns rows whose `project` is the tracking-repo root (createEscalation normalizes via `trackingProjectRoot`, `supervisor-store.ts:855`); `runConductorPass`'s `project` arg is the same tracking root the reconcile pass runs under, so the equality filter matches.
