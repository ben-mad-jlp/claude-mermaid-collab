## Blueprint — Conductor status indicator in the Bridge CommandBar

### Context / current state
- `ui/src/components/supervisor/bridge/CommandBar.tsx` renders the Bridge's top identity strip. Row 1 (`CommandBar.tsx:59-101`) shows `Bridge` + project name + `<OrchestratorLadder project={project} />` + refresh/settings buttons. There is **no** surface here that tells the operator whether the **autonomous conductor** is currently driving this project.
- The conductor enable/disable state already has a shared hook: `useConductorEnabled(project)` in `ui/src/components/supervisor/bridge/useConductorEnabled.ts:23-51`, returning `{ enabled: boolean | null, busy, setEnabled }`, seeded via `GET /api/supervisor/conductor?project=` and polled every 10s. It's currently consumed only by the interactive toggle `ConductorControl` inside `ProjectSettingsModal.tsx:215-232`.
- There is an established **read-only badge** precedent for exactly this shape: `OrchestratorLevelBadge.tsx:24-69` — a compact colored-dot indicator that self-fetches its own state, renders `null` until resolved, and carries a native `title` tooltip. The interactive control lives elsewhere; the badge only *shows*.

### The change
Add a compact, **read-only** conductor status indicator to the CommandBar Row 1, immediately after the `OrchestratorLadder`. It reuses the existing shared `useConductorEnabled` hook (read side only — never calls `setEnabled`), mirroring the `OrchestratorLevelBadge` read-only precedent. The interactive toggle stays in `ProjectSettingsModal`; this is a glance indicator, not a second control.

**1. New component: `ui/src/components/supervisor/bridge/ConductorStatusBadge.tsx`**
- `export const ConductorStatusBadge: React.FC<{ project: string }>`.
- Calls `const { enabled } = useConductorEnabled(project);` (import from `./useConductorEnabled`). Ignore `busy`/`setEnabled` — read-only.
- While `enabled === null` (not yet resolved), `return null;` — same "stay hidden until it resolves" behavior as `OrchestratorLevelBadge.tsx:53`.
- Once resolved, render an inline `<span data-testid="conductor-status-badge" data-enabled={String(enabled)}>` containing a colored dot + the label `Conductor`:
  - `enabled === true`: green dot (`bg-success-500`) + label in an active/on color; `title="Autonomous conductor is driving this project's missions"`.
  - `enabled === false`: muted gray dot (`bg-gray-400 dark:bg-gray-500`) + muted label; `title="Autonomous conductor is off — enable it in Project settings"`.
- Use the same Tailwind idiom already in this folder (`text-3xs`, `inline-flex items-center gap-1`, `shrink-0`, dark-mode pairs). Dot styled like `OrchestratorLevelBadge.tsx:63-66` (`inline-block w-2.5 h-2.5 rounded-full`), or a compact `w-2 h-2`.
- `export default ConductorStatusBadge;`.

**2. Wire it into `CommandBar.tsx`**
- Add `import { ConductorStatusBadge } from './ConductorStatusBadge';` alongside the existing `OrchestratorLadder` import (`CommandBar.tsx:17`).
- In Row 1, right after `{project && <OrchestratorLadder project={project} />}` (`CommandBar.tsx:70`), add `{project && <ConductorStatusBadge project={project} />}`. Guarding on `project` matches the ladder and keeps it from firing the hook with an empty project (the hook already early-returns on falsy project, but the guard keeps Row 1 consistent). No new `CommandBarProps` field is needed — the component fetches its own state from `project`, exactly like `OrchestratorLadder`/`OrchestratorLevelBadge` do.

**3. Tests: `ui/src/components/supervisor/bridge/ConductorStatusBadge.test.tsx`**
- Mirror the fetch-mock style of `ProjectSettingsModal.test.tsx:32-59` (a `mockFetch` that answers `/api/supervisor/conductor` with `{ project, enabled }`, assigned to `global.fetch = vi.fn(...)`; `afterEach(() => vi.restoreAllMocks())`).
- `renders green enabled indicator when conductor is on` — mock returns `{ enabled: true }`; `await screen.findByTestId('conductor-status-badge')`; assert `data-enabled === 'true'` and label text `Conductor` present.
- `renders muted indicator when conductor is off` — mock returns `{ enabled: false }`; assert badge present with `data-enabled === 'false'`.

### Design / scope notes
- Read-only by design — no click handler, no `setEnabled`. Keeps a single source of truth for *changing* the setting (`ProjectSettingsModal`), matching how `OrchestratorLevelBadge` defers mutation to `OrchestratorLadder`.
- Reuses the existing shared hook and existing REST route; no backend changes.

### ACCEPTANCE CRITERIA (positive, citable)
1. `ui/src/components/supervisor/bridge/ConductorStatusBadge.tsx` exists and exports a `ConductorStatusBadge` React component that calls `useConductorEnabled(project)` and returns `null` while `enabled === null`.
2. The badge renders an element with `data-testid="conductor-status-badge"` and a `data-enabled` attribute reflecting the boolean state, with distinct enabled (green dot) vs. off (muted dot) styling.
3. `CommandBar.tsx` imports `ConductorStatusBadge` and renders `<ConductorStatusBadge project={project} />` inside Row 1 after the `OrchestratorLadder`, guarded by `project`.
4. `ConductorStatusBadge.test.tsx` contains passing named tests `renders green enabled indicator when conductor is on` and `renders muted indicator when conductor is off`.
