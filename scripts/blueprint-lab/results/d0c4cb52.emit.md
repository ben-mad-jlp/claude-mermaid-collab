## Blueprint: REST approve route delegating to `handleMissionTool('approve_mission')`

### Context (verified)

The Missions write surface in `src/routes/supervisor-routes.ts` is a set of thin REST delegates to `handleMissionTool(...)` (from `src/mcp/mission-tools.ts`), all nested inside the guard block that begins at **`src/routes/supervisor-routes.ts:248`** (`if (url.pathname.startsWith('/api/supervisor/missions/') || url.pathname === '/api/supervisor/missions')`) and closes at **line 344**. Existing sibling sub-routes inside this block: `activate` (`:286`), `owner` (`:297`), `criteria` POST/PATCH/DELETE (`:308`–`:343`).

The `approve_mission` handler already exists at **`src/mcp/mission-tools.ts:101`**. It reads `{ project, todoId, approvedBy }`, throws `'Missing required: project, todoId'` when either is absent, calls `approveMissionAndConstitution`, broadcasts, and returns a JSON string with `{ ...result, rollup, constitutionHealth }`. No new tool logic is needed — this leaf only adds the HTTP entry point.

`jsonError(message, status)` helper is at `:56`; `handleSupervisorRoutes` is the exported entry at `:103`. `Response.json(out ? JSON.parse(out) : {})` is the established return shape for these delegates.

### Change 1 — add the route (edit `src/routes/supervisor-routes.ts`)

Insert a new POST handler for `'/api/supervisor/missions/approve'` inside the missions block, immediately after the criteria DELETE handler ends at **line 343** and before the block's closing brace at **line 344**. Mirror the `activate` delegate (`:286`–`:295`) exactly:

```ts
// Approve the mission — promotes it out of draft and locks its constitution.
// Thin delegate to handleMissionTool; verdict-setting stays MCP-only per the
// authoring-surface contract noted above (line 246).
if (url.pathname === '/api/supervisor/missions/approve' && req.method === 'POST') {
  try {
    const body = (await req.json()) as Record<string, unknown>;
    const { handleMissionTool } = await import('../mcp/mission-tools.ts');
    const out = await handleMissionTool('approve_mission', body);
    return Response.json(out ? JSON.parse(out) : {});
  } catch (err) {
    return jsonError(err instanceof Error ? err.message : 'Unknown error', 500);
  }
}
```

Rationale for matching `activate`/`create`/`delete` rather than the PATCH routes: those thin delegates rely on `handleMissionTool` to validate required fields and let the catch surface the error; `approve_mission` already throws `'Missing required: project, todoId'`, so no pre-validation is duplicated here.
