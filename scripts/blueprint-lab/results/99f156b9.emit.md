## Blueprint: Fold same-pass queued-mission promotion into `runLandedEpicSweep`

### Problem / intent
`runLandedEpicSweep` (`src/services/landed-epic-sweep.ts:186`) reconciles landed `[LAND]` leaves and GCs fully-landed epic branches, but does **not** advance the mission queue. Reconciling a `[LAND]` leaf is exactly the event that converges a mission and frees its session — yet the freed session's next queued mission isn't promoted until a *separate* `runReconcilePass` (`reconcile-pass.ts:287`) or `runConductorPass` (`conductor-pass.ts:130`) tick fires. Folding promotion into the *same* sweep pass closes that latency gap: reconcile → promote, atomically within one throttled sweep.

The promotion primitive `promoteQueuedMissions(project)` (`src/services/mission-store.ts:562`) is already idempotent DB-only (no-op for any session that already has an active mission), so running it in a third site is safe — a subsequent reconcile/conductor promotion just returns `[]`.

### Notes on ordering & idempotence
Promotion runs **after** `reconcileLandedEpics` because reconcile is what completes the `[LAND]` leaf / converges the mission that frees the session — placing it before would promote against stale active-mission state. It runs after `gcEpicBranches` too, which is harmless (GC is branch hygiene, orthogonal to mission activation). A second forced sweep over the same state promotes nothing new (`promoteQueuedMissions` is a no-op once a session has an active mission), preserving the sweep's "second pass makes zero writes" idempotence contract at the mission layer.
