## Blueprint: `validateContractForKind` + §4 strictness matrix

### Context
`src/services/diff-contract.ts` currently defines the v2 `DiffContract` type, `parseDiffContract` (fail-safe extractor), and `renderContract` (round-trip). It is a standalone leaf module (mirrors `split-decision.ts`) and is **not yet wired into any pipeline**. This leaf adds a *pure, synchronous* post-parse validator that scores an already-parsed `DiffContract` against a per-`leafKind` strictness matrix and reports two issue classes: `underspecified` (requirements too weak for the kind) and `missingField` (a required contract field is empty for the kind). No I/O, no filesystem, no wiring — same "standalone primitive" altitude as the rest of the file.

### File to edit
`src/services/diff-contract.ts` (append after `renderContract`, keeping the existing type/parse/render structure).

### Change shape

**1. Issue + result types** (new exported interfaces):
```ts
export type ContractIssueCode = 'underspecified' | 'missingField';

export interface ContractIssue {
  code: ContractIssueCode;
  /** For missingField: the empty contract field ('files' | 'tasks'). */
  field?: 'files' | 'tasks';
  /** For underspecified: the requirement kind(s) the matrix demanded. */
  requirementKinds?: DiffRequirement['kind'][];
  message: string;
}

export interface ContractValidation {
  ok: boolean;                 // issues.length === 0
  leafKind: DiffLeafKind;
  issues: ContractIssue[];
}
```

**2. The §4 strictness matrix** (new exported const) — one entry per `DiffLeafKind`, closed over `DIFF_LEAF_KINDS` via `Record<DiffLeafKind, KindStrictness>` so a new kind fails to compile until it gets a row:
```ts
interface KindStrictness {
  mustHave: DiffRequirement['kind'][];   // each listed kind must appear ≥1 time
  oneOf: DiffRequirement['kind'][];      // ≥1 requirement among these (empty ⇒ no constraint)
  requiresFiles: boolean;                // filesToCreate ∪ filesToEdit must be non-empty
}

export const KIND_STRICTNESS: Record<DiffLeafKind, KindStrictness> = {
  feature:  { mustHave: ['symbol-present', 'named-test'], oneOf: [], requiresFiles: true },
  fix:      { mustHave: ['named-test'],                   oneOf: [], requiresFiles: true },
  refactor: { mustHave: [],  oneOf: ['named-test', 'threshold'],     requiresFiles: true },
  test:     { mustHave: ['named-test'],                   oneOf: [], requiresFiles: true },
  infra:    { mustHave: [],  oneOf: ['symbol-present', 'named-test', 'threshold'], requiresFiles: false },
};
```
Rationale carried in a doc comment: `feature` must both add a symbol and cover it with a test; `fix` must ship a regression test; `refactor` must be pinned by a test or a threshold; `test` must add a named test; `infra` is loosest — any one mechanical requirement suffices, and it needn't touch files.

**3. `validateContractForKind`** (new exported pure function):
```ts
export function validateContractForKind(c: DiffContract): ContractValidation
```
Logic:
- Look up `const rule = KIND_STRICTNESS[c.leafKind]`.
- Build `const kinds = new Set(c.requirements.map(r => r.kind))`.
- For each `k` in `rule.mustHave` not in `kinds` → push `{ code: 'underspecified', requirementKinds: [k], message: \`${c.leafKind} requires a '${k}' requirement\` }`.
- If `rule.oneOf.length > 0` and none of `rule.oneOf` are in `kinds` → push one `{ code: 'underspecified', requirementKinds: rule.oneOf, message: \`${c.leafKind} requires at least one of: ${rule.oneOf.join(', ')}\` }`.
- If `rule.requiresFiles` and `c.filesToCreate.length + c.filesToEdit.length === 0` → push `{ code: 'missingField', field: 'files', message: \`${c.leafKind} must declare filesToCreate or filesToEdit\` }`.
- Always: if `c.tasks.length === 0` → push `{ code: 'missingField', field: 'tasks', message: 'contract declares no tasks' }`.
- Return `{ ok: issues.length === 0, leafKind: c.leafKind, issues }`.

Never throws (all inputs are already-normalized `DiffContract` values from `parseDiffContract`).

### Test to add
`src/services/__tests__/diff-contract.test.ts` — a new `describe('validateContractForKind (§4 strictness matrix)', ...)` block with cases:
- a fully-specified `feature` contract (symbol-present + named-test + files + tasks) ⇒ `ok: true`, `issues: []`.
- a `feature` missing the `named-test` requirement ⇒ `ok: false` with an `underspecified` issue naming `named-test`.
- a `feature` with no `filesToCreate`/`filesToEdit` ⇒ a `missingField` issue with `field: 'files'`.
- an `infra` contract with a single `threshold` requirement and no files ⇒ `ok: true` (loosest row exercised).
- `KIND_STRICTNESS` has exactly one row per `DIFF_LEAF_KINDS` member (`expect(Object.keys(KIND_STRICTNESS).sort()).toEqual([...DIFF_LEAF_KINDS].sort())`).
