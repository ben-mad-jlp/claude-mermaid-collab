## Blueprint: Serve and accept the orchestration kinds on the node-profiles route

### Context

Commit `c1f754bb` added an **orchestration-node registry** (`src/services/node-kinds.ts`) with kinds `forge | conductor | planner` and their `ORCHESTRATION_NODE_PROFILE` / `ORCHESTRATION_NODE_DESCRIPTIONS`. `LEAF_NODE_GROUPS` in `leaf-executor.ts:711-717` already declares an **`orchestration` group** whose `kinds` are `ORCHESTRATION_NODE_KINDS`, and the partition/drift-guard test (`orchestrator-routes.test.ts:195-206`) already passes for the *groups* payload.

But the `/api/orchestrator/node-profiles` route does **not** yet serve or accept these kinds:

- **GET** (`orchestrator-routes.ts:120-162`) builds `rows` only from `MATRIX_NODE_KINDS` (`orchestrator-routes.ts:27` = leaf kinds minus `summary`). So the UI renders an *Orchestration* group header with **zero editable rows**.
- **POST** (`orchestrator-routes.ts:166-196`) rejects any `kind` not in `LEAF_NODE_KINDS` (`orchestrator-routes.ts:172-174`), and its MCP guard calls `kindRequiresMcp(kind as keyof typeof NODE_PROFILE)` (`orchestrator-routes.ts:35-37, 182`) which indexes **only** `NODE_PROFILE` — an orchestration kind would read `undefined.allowedTools` and throw.

The resolvers already honor per-kind overrides for these kinds (`mission-forge.ts:366-367`, `mission-planner.ts:135-136`, `conductor-pass.ts:123` all call `resolveNodeModel`/`resolveNodeProvider`/`resolveOrchestrationEffort`, which read `node_profile_override`). The storage layer is kind-agnostic (`setNodeProfileOverride(project, kind: string, …)` in `orchestrator-config.ts:332`). **Only the route is gated.** This leaf opens the route to the three orchestration kinds.

Note: `conductor` and `planner` carry `mcp__` tools in their `ORCHESTRATION_NODE_PROFILE.allowedTools` (`node-kinds.ts:14-21, 26`) → both are MCP-forced (effective provider locked to `claude`); `forge` (`'Read Grep Glob Bash'`) is not.

### Changes

**1. `src/routes/orchestrator-routes.ts` — imports.**
Extend the existing `leaf-executor.ts` import at line 20 to also pull `ORCHESTRATION_NODE_KINDS`, `ORCHESTRATION_NODE_PROFILE`, `ORCHESTRATION_NODE_DESCRIPTIONS` (all re-exported from `leaf-executor.ts:279-284`), plus the `OrchestrationNodeKind` type.

**2. `src/routes/orchestrator-routes.ts` — merged lookups (near lines 25-37).**
Add module-level constants that unify leaf + orchestration kinds:

```ts
// The orchestration nodes (forge/conductor/planner) run ABOVE the leaf pipeline but are
// configurable in the SAME matrix — merge them into the served/accepted kind space.
const ALL_MATRIX_KINDS: (keyof typeof NODE_PROFILE | OrchestrationNodeKind)[] =
  [...MATRIX_NODE_KINDS, ...ORCHESTRATION_NODE_KINDS];
const ALL_NODE_PROFILES: Record<string, { model: string; allowedTools: string; effort: EffortLevel }> =
  { ...NODE_PROFILE, ...ORCHESTRATION_NODE_PROFILE };
const ALL_NODE_DESCRIPTIONS: Record<string, string> =
  { ...NODE_KIND_DESCRIPTIONS, ...ORCHESTRATION_NODE_DESCRIPTIONS };
// POST accepts every configurable kind (leaf kinds incl. 'summary' + orchestration kinds).
const ALL_POST_KINDS: string[] = [...LEAF_NODE_KINDS, ...ORCHESTRATION_NODE_KINDS];
```

Rewrite `kindRequiresMcp` (line 35-37) to be kind-agnostic:

```ts
function kindRequiresMcp(kind: string): boolean {
  return ALL_NODE_PROFILES[kind]?.allowedTools.includes('mcp__') ?? false;
}
```

**3. `src/routes/orchestrator-routes.ts` — GET rows (lines 126-146).**
Iterate `ALL_MATRIX_KINDS` instead of `MATRIX_NODE_KINDS`, and source `def`/`desc` from the merged maps:

```ts
const rows = ALL_MATRIX_KINDS.map((kind) => {
  const def = ALL_NODE_PROFILES[kind];
  const o = overrides[kind] ?? { model: null, effort: null, provider: null };
  const mcpForced = kindRequiresMcp(kind);
  const effectiveProvider = mcpForced ? 'claude' : (o.provider ?? projectProvider ?? 'claude');
  return {
    kind,
    desc: ALL_NODE_DESCRIPTIONS[kind],
    defaultModel: def.model,
    defaultEffort: def.effort,
    modelOverride: o.model,
    effortOverride: o.effort,
    providerOverride: o.provider,
    effectiveModel: o.model ?? def.model,
    effectiveEffort: o.effort ?? projectEffort ?? def.effort,
    effectiveProvider,
    mcpForced,
  };
});
```

The `effectiveEffort` precedence (`override → projectEffort → def.effort`) already mirrors `resolveOrchestrationEffort` (`node-provider.ts:182-188`), so orchestration rows resolve correctly with no special case.

**4. `src/routes/orchestrator-routes.ts` — POST kind validation (lines 172-173).**
Widen the whitelist and keep the message accurate:

```ts
if (!kind || !ALL_POST_KINDS.includes(kind)) {
  return jsonError(`kind must be one of: ${ALL_POST_KINDS.join(', ')}`, 400);
}
```

The MCP-forced guard at line 182 (`kindRequiresMcp(kind as …)`) now works for orchestration kinds via the rewritten helper — drop the `as keyof typeof NODE_PROFILE` cast since the param is `string`. The `providerModelMismatch` / `setNodeProfileOverride` path below is already kind-agnostic and needs no change.

**5. `src/routes/__tests__/orchestrator-routes.test.ts` — update + add tests.**

- **Update** the row-count assertion (line 166) so it accounts for the three new rows:
  ```ts
  expect(body.rows.length).toBe(
    LEAF_NODE_KINDS.filter((k) => k !== 'summary').length + ORCHESTRATION_NODE_KINDS.length,
  );
  ```
- **Add** a test `GET node-profiles serves the orchestration kinds`: asserts `body.rows` contains a row for each of `forge`, `conductor`, `planner`, that `forge.defaultModel === 'opus'` / `defaultEffort === 'high'`, and that `conductor.mcpForced === true` with `conductor.effectiveProvider === 'claude'`.
- **Add** a test `POST node-profiles accepts an orchestration kind`: POST `{ kind: 'planner', model: 'sonnet', effort: 'max' }` → 200, then GET and assert the `planner` row shows `effectiveModel === 'sonnet'` / `effectiveEffort === 'max'`.

- No changes to `node-kinds.ts`, the resolvers (`node-provider.ts`), or `orchestrator-config.ts` — storage and resolution already accept arbitrary kinds.
- Not adding a provider selector semantics change; orchestration kinds reuse the existing MCP-forced provider logic.
