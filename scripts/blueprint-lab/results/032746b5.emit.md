## Blueprint — empty-serve is not a success (productive-pass guard)

### Root cause (confirmed in `src/services/conductor-pass.ts:266-271`)
On `res.ok` the pass unconditionally stamps the plain success fingerprint (`stampConductorRun(project, missionId, fp)`). But `fp` is computed at line 234 from the **pre-invoke** state, and a conductor node can return `ok` while filing **zero** epics (LLM no-op or a swallowed `plan_mission_criterion` error). The state never moves, so `lastConductorKey === fp` forever ⇒ line 237 debounces every later tick ⇒ permanent wedge (observed live on `9688e874`: "conducted", 0 epics, stranded 4/7).

The retry machinery to escape this **already exists** (the `failPrefix`/`priorFails`/`CONDUCTOR_SERVE_RETRY_CAP` path at lines 242-244, 268-269). The bug is only that an *empty ok serve* takes the success branch instead of the fail branch. The fix routes an empty serve down the existing retry path.

Add an exported pure helper near `conductorFingerprint` (~line 59):

```ts
/** A pass "moved the mission" only if it had nothing to file (no 'discover' gaps this pass — a
 *  verify/land/building pass legitimately serves no epic) OR it actually served: at least one
 *  criterion that WAS 'discover' before the node ran now has a LIVE serving epic (action
 *  'building'). An ok node that served NOTHING for an open discover gap is an EMPTY SERVE — a
 *  no-op that must NOT stamp the success fingerprint (that is the permanent-wedge bug). */
export function conductorPassServedGaps(
  discoverIdsBefore: string[],
  actionsAfter: { id: string; action: CriterionAction }[],
): boolean {
  if (discoverIdsBefore.length === 0) return true;
  const after = new Map(actionsAfter.map((a) => [a.id, a.action]));
  return discoverIdsBefore.some((id) => after.get(id) === 'building');
}
```

(`'building'` is exactly the "live serving epic" signal — `deriveCriterionAction` returns `'building'` only when `servingEpicState==='open' && servingEpicLive`, per `mission-store.ts:922,1044-1049`.)

Replace the success/fail stamping block (current lines 263-271):

```ts
  // res.ok ALONE is not success: a node can return ok yet file no epic (LLM no-op / swallowed
  // planner error). A pass is PRODUCTIVE only if it had no discover gap to serve, or it actually
  // moved a previously-'discover' criterion to a live serving epic. An empty serve stamps the
  // bounded fail-counter (same retry path as a hard node failure) so the mission RETRIES across
  // ticks and self-heals, instead of stamping the success fp and wedging forever.
  const productive = res.ok && conductorPassServedGaps(
    discoverIdsBefore,
    listCriteriaWithActions(project, missionId).map((c) => ({ id: c.id, action: c.action })),
  );
  if (productive) {
    stampConductorRun(project, missionId, fp);
  } else {
    stampConductorRun(project, missionId, `${failPrefix}${priorFails + 1}`);
  }
  return {
    ran: true,
    reason: productive ? 'conducted' : (res.ok ? 'empty-serve' : 'node-failed'),
    missionId, modelUsed: model, escalationsRaised,
  };
```

  invokeCalls++;

test('empty-serve (ok but 0 epics filed) retries up to the cap, never debounces on the 1st', async () => {
  addWatchedProject(project);
  setConductorEnabled(project, true);
  await forgeApprovedActive();                    // one 'discover' criterion
  let calls = 0;
  const emptyServeInvoke = async () => { calls++; return { ok: true, rateLimited: false, text: 'served nothing' } as any; };
  for (let i = 0; i < CONDUCTOR_SERVE_RETRY_CAP; i++) {
    const r = await runConductorPass(project, { invoke: emptyServeInvoke });
    expect(r.ran).toBe(true);
    expect(r.reason).toBe('empty-serve');         // NOT 'debounced' — the mission RETRIES
  }
  expect(calls).toBe(CONDUCTOR_SERVE_RETRY_CAP);  // retried, did not wedge on the 1st ok
  const capped = await runConductorPass(project, { invoke: emptyServeInvoke });
  expect(capped.ran).toBe(false);
  expect(capped.reason).toBe('debounced');        // past the cap → stop respinning
  expect(calls).toBe(CONDUCTOR_SERVE_RETRY_CAP);
