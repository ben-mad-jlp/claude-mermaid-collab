## Blueprint: Enqueue instead of deactivate on create_mission / forge / approve

### Problem
Three creation/approval paths handle the one-active-per-session invariant by calling `setMissionActive(…, false)` (or, for approve, by unconditionally stealing focus with `active = 1`). A bare `active=0` mission has `queuePos = null`, so `promoteQueuedMissions` (`mission-store.ts:553`) — which requires `m.mission.queuePos != null` — will **never** promote it when the driving mission converges. Result: a newly created/approved mission that lost the focus race strands forever. The fix routes these paths through `enqueueMission` (`mission-store.ts:522`) so the mission gets a FIFO `queuePos` and re-enters rotation.

### Changes

**1. `src/mcp/mission-tools.ts` — `create_mission` handler (line 70)**

Currently:
```ts
if (sessionHasActiveMission(project, session, node.id)) setMissionActive(project, node.id, false);
```
Change to enqueue (create_mission always produces an APPROVED mission, so a queue slot is always valid):
```ts
// One-active-per-session: if this session is already driving a mission, ENQUEUE the new
// one (FIFO) so promoteQueuedMissions rotates to it — don't strand it at active=0/queuePos=null.
if (sessionHasActiveMission(project, session, node.id)) enqueueMission(project, node.id);
```
Add `enqueueMission` to the import from `../services/mission-store.js` (line 13, alongside `setMissionActive`). `setMissionActive` stays imported (still used by `set_active_mission` path? — it is used only here in create; keep it if referenced elsewhere, otherwise it remains for `activateMission`. Leave the import list otherwise intact.)

**2. `src/mcp/tools/mission-forge.ts` — `forgeMission` (lines 131–135)**

Currently:
```ts
const activate = (input.activate ?? true) && approved;
// One-active-per-session: never steal focus unless explicitly told to activate.
if (!activate || sessionHasActiveMission(project, session, missionId)) {
  setMissionActive(project, missionId, false);
}
```
Change so an **approved** mission that can't take focus is queued, while an **unapproved** mission stays inactive-and-unqueued (the queue is for approved missions only — `promoteQueuedMissions` filters `awaitingApprovalSince == null`; an unapproved mission must wait for `approve_mission`, not FIFO promotion):
```ts
const activate = (input.activate ?? true) && approved;
if (!approved) {
  // Unapproved (doc→node forge): inactive and NOT queued — it waits for approve_mission,
  // not FIFO promotion (promoteQueuedMissions skips awaitingApprovalSince rows).
  setMissionActive(project, missionId, false);
} else if (!activate || sessionHasActiveMission(project, session, missionId)) {
  // Approved but the session is already driving one (or explicit activate:false): ENQUEUE FIFO.
  enqueueMission(project, missionId);
}
```
Add `enqueueMission` to the import block (lines 26–39, alongside `setMissionActive`).

**3. `src/services/mission-store.ts` — `setMissionApproved` (lines 330–337)**

Currently unconditionally sets `active = 1`, which steals focus from a mission the session is already driving (and can leave two `active=1` rows in one session). Change to: clear the approval gate, then enqueue if the session already has an active non-terminal mission, else activate. Uses the store-local `sessionHasActiveMission` / `enqueueMission` (same file) and resolves the canonical id via `getMission`:
```ts
export function setMissionApproved(project: string, todoId: string): MissionRow {
  const m = getMission(project, todoId);
  if (!m) throw new Error(`mission not found: ${todoId}`);
  const id = m.todoId; // canonical (getMission resolves short ids)
  // Clear the approval gate first so the row reads approved before we choose active vs queued.
  openDb(project)
    .prepare('UPDATE mission SET awaitingApprovalSince = NULL, updatedAt = ? WHERE todoId = ?')
    .run(nowMs(), id);
  // One-active-per-session: don't steal focus from a mission already being driven — ENQUEUE
  // instead so promoteQueuedMissions rotates to it when the driver converges.
  const self = listMissions(project).find((x) => x.node.id === id);
  const session = self?.ownerSession ?? self?.assigneeSession ?? null;
  if (session && sessionHasActiveMission(project, session, id)) {
    return enqueueMission(project, id);
  }
  setMissionActive(project, id, true);
  return getMission(project, id)!;
}
```
Note `setMissionApproved` is defined *above* `enqueueMission`/`sessionHasActiveMission`/`listMissions` in the file — these are hoisted `function` declarations, so forward reference is fine (same pattern already used: `setMissionAbandoned` at line 345 calls `deactivateIfTerminal` defined at 458).
