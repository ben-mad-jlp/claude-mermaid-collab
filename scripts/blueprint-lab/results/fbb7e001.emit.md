## Blueprint — Add a repeatable two-mission live pin harness script

### Context / what exists

The prior leaf (`71b4b916`, commit `ff799223`) did a **manual, one-off** live measurement of the conductor-target-pin feature and wrote it up in `docs/conductor-target-live-measurement.md`. That measurement can't be re-run without a human hand-driving `curl`s. This leaf ships a **repeatable script** that automates the same live pin round-trip against the deployed sidecar, so the measurement becomes a one-command smoke test.

The live surface it drives (all real, already shipped):
- `GET /api/supervisor/conductor?project=` → `{ project, enabled, targetMissionId }` (`src/routes/supervisor-routes.ts:898-906`)
- `POST /api/supervisor/conductor` body `{ project, targetMissionId?: string|null }` → echoes `{ ok, project, enabled, targetMissionId }` (`src/routes/supervisor-routes.ts:909-942`)
- `GET /api/supervisor/missions?project=` → `{ missions: MissionSummary[] }` (`src/routes/supervisor-routes.ts:233-243`)
- Auto-clear-on-terminal-target branch it exercises: `src/services/conductor-pass.ts:113-119` (`setConductorTargetMission(project, null)`, reason `target-cleared`).

`MissionSummary` shape to read (`src/services/mission-store.ts:806-817`): `.node.id`, `.mission.active`, `.mission.awaitingApprovalSince`, `.rollup.status`. The actionable predicate must mirror `conductor-pass.ts:101-104` exactly: `mission.active && awaitingApprovalSince == null && status ∉ {unapproved, abandoned, converged}`, using `rollup.status` (both are `deriveMissionStatus`, `mission-store.ts:272` / `:884`). Terminal = `rollup.status ∈ {abandoned, converged}`.

Idiom to match: `scripts/smoke-coordinator-live.ts` (`check()`/`pass`/`fail`/`process.exit(fail===0?0:1)`) and the exported-pure-helpers + `bun:test` pattern of `scripts/parent-epic-under-mission.ts` ↔ `scripts/__tests__/parent-epic-under-mission.test.ts`.

### Change 1 — `scripts/two-mission-pin-harness.ts` (new)

A Bun script (`bun run scripts/two-mission-pin-harness.ts`) with **exported pure helpers** (testable offline) plus an `import.meta.main` live driver.

Exported symbols:

1. `interface HarnessMission { id: string; title: string; active: boolean; awaitingApprovalSince: number | null; status: string }` — the normalized slice pulled from a `MissionSummary`.

2. `function normalizeMissions(missions: any[]): HarnessMission[]` — maps each raw `MissionSummary` to `{ id: m.node.id, title: m.node.title, active: m.mission.active, awaitingApprovalSince: m.mission.awaitingApprovalSince, status: m.rollup.status }`.

3. `function classifyMissions(missions: HarnessMission[]): { actionable: HarnessMission[]; terminal: HarnessMission[] }` — actionable = `active && awaitingApprovalSince == null && !['unapproved','abandoned','converged'].includes(status)`; terminal = `['abandoned','converged'].includes(status)`. Comment cites `conductor-pass.ts:101-104` as the mirrored predicate.

4. `function selectPinPair(c): { pinA: HarnessMission; pinB: HarnessMission | null } | { skip: string }` — `pinA` = `c.actionable[0]`; `pinB` = `c.terminal[0] ?? null`. If no actionable mission, return `{ skip: 'no-actionable-mission' }` (never seed fake missions — per prior leaf's non-goal in `docs/conductor-target-live-measurement.md` Finding 1).

5. `interface HarnessResult { pass: number; fail: number; skipped: string | null; checks: Array<{ name: string; ok: boolean; detail?: string }> }`

6. `async function runHarness(opts: { baseUrl: string; project: string; tickTimeoutMs?: number; pollIntervalMs?: number; fetch?: typeof fetch }): Promise<HarnessResult>` — the sequence, with a `try/finally` that **always restores** the conductor's original `targetMissionId` (idempotency/repeatability is the point):
   - GET `/api/supervisor/conductor` → capture `originalTarget` (baseline).
   - GET `/api/supervisor/missions` → `normalizeMissions` → `classifyMissions` → `selectPinPair`. On `{skip}`, set `skipped`, restore, return early (exit code 0 — a skip is not a failure, matching Finding 1's "no second live competitor" reality).
   - POST pin `pinA.id`; check echo `targetMissionId === pinA.id`.
   - GET; check pin holds `=== pinA.id`.
   - Poll GET every `pollIntervalMs` (default 5000) up to `tickTimeoutMs` (default 90000, covers the 30s orchestrator tick, `orchestrator-live.ts` interval); check pin **still** `=== pinA.id` after ≥1 real tick (no drift).
   - If `pinB`: POST pin `pinB.id`; check echo `=== pinB.id`; then poll GET until `targetMissionId === null` within `tickTimeoutMs`; check auto-cleared (exercises `conductor-pass.ts:113-119`). If no `pinB`, record a skipped-sub-check note (log it — no silent cap).
   - `finally`: POST `{ targetMissionId: originalTarget }` to restore the live conductor to exactly its pre-run state.
   - Helper `postConductor` / `getConductor` / `getMissions` inner fns use `opts.fetch ?? fetch` so tests can inject a fake.

7. `import.meta.main` block: read `baseUrl` from `http://localhost:${process.env.MERMAID_PORT ?? 9002}`, `project` from `process.env.MERMAID_PROJECT ?? process.cwd()` (or first CLI arg), call `runHarness`, print each check (`✅`/`❌`) and the pass/fail summary line, `process.exit(result.fail === 0 ? 0 : 1)`.

### Change 3 — `scripts/__tests__/two-mission-pin-harness.test.ts` (new)

`bun:test` unit tests (no live server — deterministic), importing from `../two-mission-pin-harness.ts`:
- `classifyMissions` splits a fixture of mixed missions into the right actionable/terminal buckets (incl. an `unapproved` and an `awaitingApprovalSince != null` correctly excluded from actionable).
- `selectPinPair` returns `{skip:'no-actionable-mission'}` when none actionable; returns `pinA` = first actionable and `pinB` = first terminal otherwise; `pinB === null` when no terminal mission exists.
- `runHarness` against a **fake fetch** stub that models the live server (baseline target null → pin A echoes A → tick GETs return A → pin B echoes B → a later GET returns null): asserts the check sequence all-passes, that the auto-clear check fired, and that the **final restore POST** carried `originalTarget` (assert the stub recorded a trailing POST with the baseline target) — proving idempotency.

### Acceptance criteria (positive, citable)

1. `scripts/two-mission-pin-harness.ts` exists and exports `classifyMissions`, `selectPinPair`, and `runHarness` (point at their `export function`/`export async function` lines).
2. `classifyMissions` computes `actionable` using the `active && awaitingApprovalSince == null && !['unapproved','abandoned','converged'].includes(status)` predicate mirroring `conductor-pass.ts:101-104` (point at the predicate line).
3. `runHarness` performs the POST-pin-A → GET-verify → poll-hold-across-tick → POST-pin-B → poll-auto-clear-to-null sequence and, in a `finally`, POSTs the captured `originalTarget` back (point at the restore POST line).
4. `runHarness` returns `{ skip: 'no-actionable-mission' }`-driven early exit (via `selectPinPair`) rather than fabricating missions when no actionable mission is live (point at the `skip` branch).
5. `scripts/__tests__/two-mission-pin-harness.test.ts` contains a passing `runHarness`-against-fake-fetch test asserting the trailing restore POST carried the baseline target (point at the test's assertion).
6. `package.json` `"scripts"` has a `"smoke:conductor-pin"` entry invoking the harness (point at the line).

### Out of scope
- Seeding/forging a second real actionable mission into the live store to manufacture an A-vs-B race (explicitly a non-goal, `docs/conductor-target-live-measurement.md` Finding 1).
- Forcing a self-destructive `POST /api/supervisor/deploy {force:true}` from within the harness.
- Any change to conductor selection/pin logic in `conductor-pass.ts` or the route handlers — this leaf only *measures* them.
