## Blueprint: Live-mode preflight of the real conductor

### Context / problem

`scripts/smoke-conductor-pin-live.ts` has two transports (`scripts/smoke-conductor-pin-live.ts:44-45`): in-process (default) and live HTTP (`--base-url`). In **live mode**, Phase 3 relies on the *deployed* server's own orchestrator conductor timer to advance the pinned mission, so it deliberately does **not** call `startOrchestrator()`:

```ts
// scripts/smoke-conductor-pin-live.ts:284-287
if (!liveMode) {
  startOrchestrator!();
  started = true;
}
```

The gap: nothing verifies the deployed server's conductor timer is actually *running*. Remote/headless collab servers boot with `MERMAID_AUTO_START_COORDINATOR=0` (no daemon — see `bin`/launch path), so `getOrchestratorHealth().running` is `false` and **nothing will ever move mission A**. Today that surfaces only as the Phase 3 assertion `mission A advanced across >= 2 distinct conductor ticks` failing after a ~2-minute poll (`scripts/smoke-conductor-pin-live.ts:289, 323-327`) with an opaque transcript — the script *silently skips* the start and lets the real symptom masquerade as a timing failure. It should **preflight** the deployed conductor and **fail loudly** with the root cause up front.

### The signal to check

`GET /api/orchestrator/health` (`src/routes/orchestrator-routes.ts:298-316`) returns `getOrchestratorHealth()` (`src/services/orchestrator-live.ts:518-546`), whose `running` field is `isOrchestratorRunning()` = `timer !== null` (`src/services/orchestrator-live.ts:514-516`). `startOrchestrator()` starts `timer`, `summaryTimer`, and `conductorTimer` together atomically (`src/services/orchestrator-live.ts:445-485`), so `running === true` is a sound proxy for "the conductor heartbeat is live." The route also exposes `tickMs` and `lastTickAt` for a richer diagnostic.

### Change shape — single file: `scripts/smoke-conductor-pin-live.ts`

**1. Add a preflight function** (place it after the `get()` helper, ~line 108, near the other route helpers). It runs the check only in live mode, records a positive `check(...)` line, and **throws** on failure so the run aborts loudly through `main`'s `finally` cleanup:

```ts
// Live-mode preflight: the deployed server — not this script — owns the conductor
// timer. If it booted with MERMAID_AUTO_START_COORDINATOR=0 the conductor never ticks
// and mission A can never advance. Prove the daemon is live BEFORE Phase 3 polls, and
// fail loudly with the root cause instead of a 2-minute opaque "did not advance" timeout.
async function preflightLiveConductor(): Promise<void> {
  const health = await get('/api/orchestrator/health');
  const running = health.body?.running === true;
  check(
    'deployed orchestrator conductor timer is running (GET /api/orchestrator/health running===true)',
    running,
    `status=${health.status} health=${JSON.stringify(health.body)}`,
  );
  if (!running) {
    throw new Error(
      `live preflight FAILED: deployed server at ${baseUrl} reports orchestrator running=${health.body?.running}. ` +
      `The conductor timer is not ticking (likely launched with MERMAID_AUTO_START_COORDINATOR=0), so the pinned ` +
      `mission can never advance. Start the daemon on the target server, then re-run.`,
    );
  }
}
```

**2. Call it in Phase 3**, replacing the current live-mode skip so live mode preflights instead of doing nothing (`scripts/smoke-conductor-pin-live.ts:284-287`):

```ts
if (liveMode) {
  await preflightLiveConductor();
} else {
  startOrchestrator!();
  started = true;
}
```

**3. Update the Phase 3 log line and the header docstring** to state that live mode preflights the deployed conductor. Adjust the module docstring (`scripts/smoke-conductor-pin-live.ts:22-24`) which currently says Phase 3's start/stop is "skipped/gated" — change to note it is preflighted-and-required in live mode. Update the `Phase 3` `log(...)` at `scripts/smoke-conductor-pin-live.ts:280` to mention the preflight.
