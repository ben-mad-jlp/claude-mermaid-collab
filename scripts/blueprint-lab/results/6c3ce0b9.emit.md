## Implementation Blueprint — Enforce a mission-creation ceiling in the mission store + creation paths

### Problem & chokepoint
Every mission-creation path funnels through **one** function — `upsertMission` in `src/services/mission-store.ts:282` — which attaches the durable control row that makes a `[MISSION]` graph node a *real* mission:
- `create_mission` handler → `mission-tools.ts:65` calls `upsertMission`
- `forgeMission` → `mission-forge.ts:123` calls `upsertMission`
- `forge_mission_from_doc` → `forgeMissionFromDoc` → `forgeMission` (covered transitively)
- mission-intake pass → `forgeMissionFromDoc` → `forgeMission` (covered; intake also has its own separate `MAX_PENDING` unapproved-draft cap at `mission-intake.ts:45`)

There is currently **no bound** on how many live missions a project can accrue. The fix places the **enforced backstop in `upsertMission`** (no path can bypass it, per CLAUDE.md "enforce in code") plus a **pre-flight check in the two node-creating callers** so a rejected create never strands an orphan `[MISSION]` node (both callers do `addSessionTodo` *before* `upsertMission`).

"Ceiling" counts **live (non-terminal)** missions — `!isMissionTerminal(m.mission)` — so a project that converged/abandoned 30 missions can still start new ones; only concurrently-in-play missions consume the budget. This reuses the existing `isMissionTerminal` (`mission-store.ts:42`) and `listMissions` (`mission-store.ts:914`).

```ts
/** Default per-project cap on LIVE (non-terminal) missions. A safety backstop against
 *  runaway mission creation; overridable per deploy via config/env MAX_LIVE_MISSIONS. */
export const DEFAULT_MAX_LIVE_MISSIONS = 12;

/** Resolve the live-mission ceiling: env/config MAX_LIVE_MISSIONS if a positive int, else default. */
export function missionCreationCeiling(): number {
  const n = Number(getConfig('MAX_LIVE_MISSIONS', '') || 0);
  return Number.isFinite(n) && n > 0 ? Math.floor(n) : DEFAULT_MAX_LIVE_MISSIONS;
}

/** Thrown when creating another mission would exceed the live-mission ceiling. Typed so
 *  callers can surface it distinctly (creation paths translate it to a user-facing message). */
export class MissionCeilingError extends Error {
  constructor(public readonly live: number, public readonly ceiling: number) {
    super(`mission-creation ceiling reached: ${live} live missions >= ceiling ${ceiling} `
      + `(converge, abandon, or delete a mission, or raise MAX_LIVE_MISSIONS)`);
    this.name = 'MissionCeilingError';
  }
}

/** Count LIVE (non-terminal) missions in a project. Terminal (converged/abandoned) missions
 *  do NOT count against the ceiling. `excludeTodoId` omits one node (defensive; a node being
 *  created has no control row yet so listMissions already skips it). */
export function countLiveMissions(project: string, excludeTodoId?: string): number {
  return listMissions(project).filter(
    (m) => m.node.id !== excludeTodoId && !isMissionTerminal(m.mission),
  ).length;
}

/** Enforce the live-mission ceiling; throws MissionCeilingError when at/over it. The single
 *  guard both upsertMission (backstop) and the node-creating callers (pre-flight) call. */
export function assertMissionCeilingNotExceeded(project: string, excludeTodoId?: string): void {
  const ceiling = missionCreationCeiling();
  const live = countLiveMissions(project, excludeTodoId);
  if (live >= ceiling) throw new MissionCeilingError(live, ceiling);
}
```

### Notes / edge behavior
- Idempotent re-`upsertMission` on an existing mission never trips the ceiling (early return precedes the guard) — safe for the many `upsertMission(project, id)` twice-call sites.
- Default `12` is well above what any single existing test creates (max ~2–3 live missions per isolated tmpdir project), so no existing test regresses.
- The mission NODE created by `create_mission`/`forgeMission` has no control row at pre-flight time, so it is naturally excluded from the count (and `excludeTodoId` guards the backstop defensively).

### Out of scope
- Changing the mission-intake `MAX_PENDING` unapproved-draft cap (separate, narrower gate).
- A UI affordance to raise the ceiling or surface "ceiling reached" (the typed error + tool exception propagation is the surface).
