# Blueprint: wire the autonomous LAND path (conductor actor + ownership-gated)

## Problem statement

The server-side authority is complete and correct:
- `land_epic` (`src/mcp/setup.ts:1957–2007`) accepts `actor:'conductor'` + `session`, runs `checkOwnership` (`src/services/land-authority.ts:143`), and refuses foreign/bucket epics fail-closed.
- `escalation_list` (`src/mcp/setup.ts:1932`) already exists as an MCP tool.
- The conductor node runs `permissionMode: 'bypassPermissions'` (`src/services/conductor-pass.ts:137`), so the human-classifier permission gate is genuinely bypassed — the ONLY remaining gate is the server's ownership + safety proof.

But the conductor **node** cannot actually land today:
1. `CONDUCTOR_ALLOWED_TOOLS` (`src/services/conductor-pass.ts:33–40`) omits `mcp__mermaid__escalation_list` — the node can't discover the open `epic-ready-to-land` escalation, and `land_epic` **requires** an `escalationId`.
2. `buildConductorPrompt` step 5 (`src/services/conductor-pass.ts:75–77`) says "confirm `epic_land_readiness`, then `land_epic`" but never tells the node to find the escalation, pass `actor:'conductor'`, or pass a `session`.
3. The mission's owner session is never threaded into the prompt. `checkOwnership` Rule 5 (`src/services/land-authority.ts:211`) compares `mission.ownerSession !== actor.session`, so the node MUST pass the mission todo's exact `ownerSession` or every land is refused `foreign-mission`. The node cannot derive this reliably; the pass must thread it deterministically (`ownerSession` is on the `MissionSummary` from `listMissions`, `src/services/mission-store.ts:817`).

All three changes live in one file: `src/services/conductor-pass.ts`. Plus one new test in `src/services/__tests__/conductor-pass.test.ts` (7 → 8 tests).

Add a helper mirroring `getMissionTitle` (`conductor-pass.ts:148–151`):

```ts
/** The session that OWNS this mission (mission↔session tie) — threaded to the conductor node so its
 *  land_epic passes actor:'conductor' + the exact session checkOwnership Rule 5 compares against. */
function getMissionOwnerSession(project: string, missionId: string): string | null {
  const m = listMissions(project).find((x) => x.node.id === missionId);
  return m?.ownerSession ?? null;
}
```

Use `ownerSession` specifically (NOT `assigneeSession`) because `checkOwnership` Rule 5 (`land-authority.ts:211`) compares `mission.ownerSession` exactly. `listMissions` is already imported (`conductor-pass.ts:13`).
