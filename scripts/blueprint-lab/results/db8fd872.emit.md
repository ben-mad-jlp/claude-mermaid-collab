# Implementation Blueprint

## Goal
Make `scripts/smoke-conductor-pin-live.ts` actually exercise the criterion-verdict → convergence → lazy-clear path **in live mode** by driving the `set_mission_criterion` MCP tool over the deployed server's `/mcp` Streamable-HTTP JSON-RPC endpoint — instead of the current dead-end that records a hard failure and skips Phase 4.

## Why this is the right seam
- Setting a criterion verdict is **deliberately not** a REST route — `src/routes/supervisor-routes.ts:249-252` keeps verdict-setting steward/MCP-only to preserve maker≠checker independence. So there is no `/api/supervisor/...` surface, and the current `markCriterionMet` live branch (`scripts/smoke-conductor-pin-live.ts:140-148`) correctly notes this, then gives up.
- But the deployed server **does** expose the verdict setter over MCP: `set_mission_criterion` is declared in `src/mcp/mission-tools.ts:38` and handled at `src/mcp/mission-tools.ts:209-222`, calling `setCriterionVerdict`/`setCriterionMet`. The server routes `POST /mcp` → `handleMCPRequest` (`src/server.ts:512-514` → `src/mcp/http-handler.ts:47`), a JSON-RPC Streamable-HTTP transport that returns `Response.json(...)` inline (`src/mcp/http-transport.ts:134-140`) and threads a session via the `Mcp-Session-Id` header (`src/mcp/http-handler.ts:149-157`).
- The mission was created over the SAME deployed server via `POST /api/supervisor/missions` (Phase 1), so its criterion row lives in that server's store; the in-process `/mcp` tool mutates the same store. Convergence then relies on the deployed server's own conductor timer to lazily clear the pin — exactly what Phase 4/5 already poll for over HTTP.
- Loopback is auth-exempt and `/mcp*` is origin-guard-exempt (`src/server.ts:480-481`), so the default `--base-url http://localhost:9002` needs no token.

## Change shape — single file: `scripts/smoke-conductor-pin-live.ts`

### 1. Add an MCP JSON-RPC fetch helper (near the existing `route`/`post`/`get` helpers, ~line 108)
```ts
// Drive one JSON-RPC message at the deployed server's /mcp Streamable-HTTP endpoint.
// Threads the Mcp-Session-Id header so the initialize→initialized→tools/call handshake
// stays on one session. Live mode only (baseUrl non-null).
async function mcpFetch(body: unknown, sessionId: string | null): Promise<{ status: number; sessionId: string | null; json: any }> {
  const res = await fetch(new URL('/mcp', baseUrl!), {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      Accept: 'application/json, text/event-stream',
      ...(sessionId ? { 'Mcp-Session-Id': sessionId } : {}),
    },
    body: JSON.stringify(body),
  });
  const sid = res.headers.get('mcp-session-id');
  const json = await res.json().catch(() => null); // 202 for notifications has no body
  return { status: res.status, sessionId: sid, json };
}
```

### 2. Replace the `liveMode` branch body of `markCriterionMet` (`scripts/smoke-conductor-pin-live.ts:140-149`)
Delete the hardcoded-`false` check + `return false`; drive the verdict over `/mcp` instead:
```ts
if (liveMode) {
  // 1) initialize (no session id yet) → server mints Mcp-Session-Id
  const init = await mcpFetch(
    { jsonrpc: '2.0', id: 1, method: 'initialize',
      params: { protocolVersion: '2024-11-05', capabilities: {}, clientInfo: { name: 'smoke-pin', version: '0' } } },
    null,
  );
  const sid = init.sessionId;
  check('MCP /mcp initialize handshake', init.status === 200 && !!init.json?.result?.serverInfo && !!sid,
    `status=${init.status} sid=${sid}`);
  // 2) notifications/initialized (202, no body)
  await mcpFetch({ jsonrpc: '2.0', method: 'notifications/initialized' }, sid);
  // 3) tools/call set_mission_criterion — the verdict setter with no REST surface
  const call = await mcpFetch(
    { jsonrpc: '2.0', id: 2, method: 'tools/call',
      params: { name: 'set_mission_criterion',
        arguments: { project, criterionId, met, evidence: 'smoke: synthetic convergence', verifiedBy: 'smoke-pin-live' } } },
    sid,
  );
  const text = call.json?.result?.content?.[0]?.text;
  let parsed: any = null; try { parsed = JSON.parse(text); } catch { /* non-JSON */ }
  const ok = call.status === 200 && !call.json?.result?.isError
    && parsed?.criterionId === criterionId && parsed?.met === met;
  check('criterion verdict driven over deployed /mcp (set_mission_criterion)', ok,
    `status=${call.status} resp=${JSON.stringify(call.json?.result ?? call.json)}`);
  return ok;
}
```
`markCriterionMet(project, criterionId, met)` keeps its signature; returning `ok` lets the existing Phase-4 `if (criterionSet) { ... }` block (`scripts/smoke-conductor-pin-live.ts:278-297`) run its convergence assertion + lazy-clear poll in live mode. The `else` branch's "skipped in live mode" log becomes unreachable for the happy path — leave it as the failure fallback.

Passing `evidence` + `verifiedBy` routes through the richer `setCriterionVerdict` path (`src/mcp/mission-tools.ts:216-217`), a faithful maker≠checker verdict rather than a bare `met` flip.

## Out of scope
- No production `src/` behavior change — verdict-setting stays MCP-only by design; this only teaches the smoke to use the MCP wire.
- No new REST route for verdicts (would break maker≠checker).
- Remote (non-loopback) `--base-url` auth-token plumbing is untouched.
