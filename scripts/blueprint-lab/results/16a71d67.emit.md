## Blueprint: Single `resolvedPort()` + config.json port key

### Problem
The server port is resolved by **duplicated, drifting one-liners** — `parseInt(process.env.PORT || '9002', 10)` appears in ~10 modules, plus `validatePort()` and `PORT_REQUEST` in `src/config.ts`. None of them honor `~/.mermaid-collab/config.json`, so a user can set `XAI_API_KEY` in the Secrets UI but has no config-file way to pin the port. This leaf introduces **one resolver** that also reads a `"PORT"` config.json key, and routes every call site through it.

### Where the resolver belongs
`src/services/config-file.ts` — the pure-fs, sqlite-free layer (`src/services/config-file.ts:1-20` docstring). It is already importable from *everything*: the vitest-loaded import-light modules, `src/config.ts`, and the stdio MCP tools. `getConfig()` there (`src/services/config-file.ts:39-45`) already implements the exact precedence we want (non-empty `process.env` → non-empty config.json string → fallback), so `resolvedPort()` composes on top of it — the `"PORT"` config.json key comes for free.

### Change 1 — define `resolvedPort()` (`src/services/config-file.ts`)
Add after `getConfig` (near line 45):

```ts
/**
 * The single source of truth for the server's HTTP port. Precedence matches
 * getConfig: non-empty process.env.PORT → config.json "PORT" (string) → 9002.
 * Parsed as int; a non-numeric / out-of-range value falls back to 9002. The
 * literal 0 is preserved (OS-assigned ephemeral-port request — see PORT_REQUEST).
 * NOTE: a config.json PORT must be a STRING ("9002") — getConfig ignores
 * non-string file values (see config-service.test.ts "non-string file value…").
 */
export function resolvedPort(): number {
  const raw = getConfig('PORT', '9002') ?? '9002';
  const n = Number.parseInt(raw, 10);
  return Number.isFinite(n) && n >= 0 && n <= 65535 ? n : 9002;
}
```

Also update the stale line in the module docstring (`src/services/config-file.ts:19`) — `config.ts`'s PORT constant is now sourced here, so drop/soften "config.ts's frozen constants are intentionally NOT migrated here yet."

### Change 3 — collapse `src/config.ts` onto the resolver
- Add import at top: `import { resolvedPort } from './services/config-file.js';` (config-file pulls in only node builtins → no cycle, no sqlite).
- Delete `validatePort()` (`src/config.ts:21-39`) and set `PORT: resolvedPort()` (`src/config.ts:53`). (Behavior shift: an invalid PORT now falls back to 9002 instead of throwing at startup — strictly more robust, and matches `getConfig`'s tolerant philosophy. No test asserts the throw.)
- Replace `PORT_REQUEST` (`src/config.ts:91-93`) with `export const PORT_REQUEST = resolvedPort();` — `resolvedPort` already preserves the literal `0`, which was `PORT_REQUEST`'s only reason to differ from `PORT`. Keep the export (consumed at `src/server.ts:447`).

Downstream consumers of `config.PORT` (`src/services/leaf-executor.ts:2026,2069,2163`, `src/services/conductor-pass.ts:131`, `src/routes/pair-routes.ts:73`, `src/mcp/tools/mission-planner.ts:144`) need **no edit** — they inherit the config.json key automatically.

### Change 5 — tests (`src/services/__tests__/config-service.test.ts`)
Add a `describe('resolvedPort')` block (the harness already stubs `MERMAID_CONFIG_PATH` + resets cache in `beforeEach`, `src/services/__tests__/config-service.test.ts:10-21`). Cover the four contract points:
- **default** — no env, `{}` file → `resolvedPort()` returns `9002`.
- **config.json PORT key honored** — write `{ PORT: '9100' }`, no env → returns `9100` (this proves the new key).
- **env wins** — file `{ PORT: '9100' }` + `process.env.PORT='9200'` → returns `9200` (clean up env in `finally`).
- **invalid falls back** — `process.env.PORT='not-a-port'` → returns `9002`.

Import `resolvedPort` from `'../config-service'` alongside the existing `getConfig` import (`src/services/__tests__/config-service.test.ts:5`).

### Acceptance criteria (positive, citable)
1. `resolvedPort()` is exported from `src/services/config-file.ts` and reads the config.json `"PORT"` key via `getConfig`.
2. `resolvedPort` is re-exported from `src/services/config-service.ts`.
3. `src/config.ts` `PORT` and `PORT_REQUEST` are both assigned from `resolvedPort()`; `validatePort()` is gone.
4. Every module in the Change-4 table calls `resolvedPort()` instead of an inline `parseInt(process.env.PORT …)`.
5. `config-service.test.ts` has a `resolvedPort` test asserting the config.json `PORT` key is honored when env is unset.

### Out of scope
- `HOST` resolution (still `process.env.MERMAID_BIND_HOST ?? process.env.HOST`).
- Non-`src/` occurrences: `scripts/deploy.ts` (uses `MERMAID_PORT`), `spike/`, `bin/mermaid-collab.ts`, `extensions/vscode`, and test fixtures (`wsl/__tests__`, `orchestrator-routes.test.ts`) keep their literals.
- Accepting a **numeric** JSON `PORT` value (kept string-only to match `getConfig` semantics).
