## Blueprint — Rule 0.5: acquit output-artifact content assertions

### Problem
`src/services/criteria-citability.ts` runs its rule ladder in `classifyCriterion` (lines 237–273) as: Rule 0 acquit-on-citation → Rule 1 out-of-diff → Rule 1.5 verification-command acquit → Rule 2 `convictOnCommandResult` → Rule 3 absence. A criterion that asserts a readable property of a *produced artifact* ("`results/report.md` contains a `## GATE` section and the gate passes", "`score.json` records the match rate") is the most falsifiable kind — the reviewer reads the produced file — yet it gets convicted: `convictOnCommandResult` (lines 127–160) fires on the result-predicate noun+verb, and if the artifact is cited with a line (`report.md:12`) Rule 1 (lines 95–124) convicts it as out-of-diff because a runtime output is never in the declared *code* change-set.

### Fix — one new acquit, placed as Rule 0.5 (before Rule 1)

**File: `src/services/criteria-citability.ts`**

1. **Add a helper `assertsOutputArtifactContent(text)`** immediately before `classifyCriterion` (after `namesVerificationCommand`, ~line 235). It acquits iff the criterion (a) names a concrete artifact filename `name.ext` whose extension is an *artifact* extension (allowlist — source-code extensions are excluded by construction), AND (b) is positive-only (does not trip `convictOnAbsence`):

```ts
/** Rule 0.5 — ACQUIT on an output-artifact content assertion. A criterion asserting a
 *  POSITIVE, readable property of a concrete PRODUCED artifact file (a report/data/log
 *  file, NOT source code) is the most falsifiable criterion shape: the review reads the
 *  produced file. The artifact is a RUNTIME OUTPUT, so it is never in the declared code
 *  change-set — this acquit therefore MUST precede the out-of-diff conviction (Rule 1).
 *  Guards keep it tight:
 *   - a concrete artifact filename (name.ext, ext ∈ the artifact allowlist) must be present,
 *     so a suite-wide "tests pass" (no file) is never acquitted;
 *   - positive-only (no absence pattern), so "config.json unchanged" is never masked;
 *   - the allowlist carries NO source-code extension, so a `foo.ts:1` citation is never
 *     acquitted here — it falls through to the ordinary citation rules.
 *  Returns kind:'command-result' so uncitedCriteriaAreAllCommandResults and the review-time
 *  grounding defer honour it identically to a verification-command acquit. */
const ARTIFACT_FILE_RE =
  /(?:^|[\s`('"[,\/])([\w.@-]+(?:\/[\w.@-]+)*\.(?:md|markdown|json|ndjson|jsonl|csv|tsv|log|txt|xml|ya?ml|out|report))\b/i;

function assertsOutputArtifactContent(text: string): boolean {
  if (!ARTIFACT_FILE_RE.test(text)) return false;
  // positive-only: an absence about an artifact ("config.json unchanged") asserts a negative
  // no read confirms — do not acquit it here; let Rule 3 convict.
  if (convictOnAbsence(text).uncitable) return false;
  return true;
}
```

2. **Wire it into `classifyCriterion`** between Rule 0 (line 243–245) and Rule 1 (line 247–251):

```ts

  // conviction — a produced artifact is a runtime output, never in the code change-set).
  if (assertsOutputArtifactContent(text)) {
    return { text, citable: true, kind: 'command-result' };
```

### Why each guard holds
- **Concrete filename required** — `ARTIFACT_FILE_RE` needs `name.ext`; "tests pass" / "the whole test suite passes" have none, so they fall through to `convictOnCommandResult` and stay `command-result`.
- **Positive-only** — `config.json unchanged` matches the filename regex but trips `convictOnAbsence` ("unchanged"), so `assertsOutputArtifactContent` returns false and Rule 3 convicts it `absence`.
- **Source excluded** — `.ts/.tsx/.js/.py/…` are absent from the allowlist, so `src/services/other.ts:42` is not acquitted by 0.5 and Rule 1 still convicts it out-of-diff.
- **`kind:'command-result'`** — matches Rule 1.5's contract so `uncitedCriteriaAreAllCommandResults` (lines 315–322) and the review-time grounding defer treat it identically.
