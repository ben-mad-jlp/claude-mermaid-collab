## Blueprint: Live two-mission pin measurement against the deployed app

### Context (what "pin" and "measure" mean here)
The **conductor pin** is `watched_project.conductorTargetMissionId` — a per-project explicit selection of *which* mission the autonomous conductor drives when two or more missions are active at once. Its whole reason for existing is disambiguation: `src/services/conductor-pass.ts:110-126` resolves *exactly* the pinned mission and **never** falls back to another. The store round-trip lives at `src/services/supervisor-store.ts:483-493` (`getConductorTargetMission`/`setConductorTargetMission`), and the deployed HTTP surface is `GET`/`POST /api/supervisor/conductor` at `src/routes/supervisor-routes.ts:896-938`.

This leaf's job is a **live smoke/measurement harness** that exercises that pin *end-to-end against a running (deployed) server on :9002* with **two** active missions — proving the HTTP write→shared-DB→conductor-selection path picks the pinned mission and only the pinned mission. The in-isolation logic is already unit-tested (`src/services/__tests__/conductor-pass.test.ts:107-190`); the new value is the *live* path against the real server, following the existing `scripts/smoke-*-live.ts` pattern (`scripts/smoke-coordinator-live.ts`, `scripts/smoke-coordinator-mcp.ts`).

### File to create
`scripts/smoke-conductor-pin-live.ts` — a `bun run` script (no MCP framework), mirroring `scripts/smoke-coordinator-live.ts`'s structure: header docblock (`Run: bun run scripts/smoke-conductor-pin-live.ts`, "Requires the HTTP API server running on :9002"), a `check(name, ok, detail)` helper (pass/fail counters), a `try/finally` with cleanup, and `process.exit(fail === 0 ? 0 : 1)` at the end.

### Exact shape

**Base URL + health gate.** Use `const BASE = process.env.MERMAID_BASE_URL ?? 'http://localhost:9002'`. First `fetch(`${BASE}/api/health`)`; if it does not return ok, print a clear "deployed server not reachable on :9002 — start it first" message and `process.exit(1)` (fail-closed — a live measurement must not silently pass when there is nothing to measure).

**Do NOT override `MERMAID_SUPERVISOR_DIR`.** The measurement's correctness depends on the in-process `runConductorPass` reading the *same* supervisor DB the HTTP route wrote to. Both default to the real global supervisor dir, so a pin POSTed over HTTP is visible to the in-process pass. (This is the deliberate opposite of `smoke-coordinator-live.ts`, which isolates the dir precisely because it does *not* need to observe the server's DB.)

**Seed (in-process, throwaway temp project).**
- `const project = mkdtempSync(join(tmpdir(), 'conductor-pin-smoke-'))`.
- `addWatchedProject(project)` from `../src/services/supervisor-store` — required because the conductor setters/route are UPDATE-only (`src/routes/supervisor-routes.ts:907-908`); an unwatched project's POST would no-op.
- Forge **two** approved+active missions via `forgeMission` (`../src/mcp/tools/mission-forge`, returns `{ missionId }`, defaults `approved:true`/`activate:true` per `src/mcp/tools/mission-forge.ts:122-128`):
  - `missionA = await forgeMission(project, { session: 's1', title: 'Pin smoke: mission A', criteria: ['criterion A'] })` — this is the first-active (unpinned fallback) mission.
  - `missionB = await forgeMission(project, { session: 's1', title: 'Pin smoke: mission B', criteria: ['criterion B'] })`.

**`async function measurePinRoundTrip(project, missionB)` — the live deployed-app round-trip.** All via `fetch(`${BASE}/api/supervisor/conductor`, …)`:
1. `POST { project, targetMissionId: missionB }` → assert `res.ok` and body `targetMissionId === missionB`.
2. `GET ?project=<enc>` → assert body `targetMissionId === missionB` (persisted through the deployed server's DB).
3. `POST { project, enabled: true }` (no targetMissionId) → GET → assert `targetMissionId` still `=== missionB` **and** `enabled === true` (enable alone must not clear the pin — mirrors `conductor-target-route.test.ts:63-73`).
4. `POST { project, targetMissionId: null }` → GET → assert `targetMissionId === null` (clear).
Each step is a `check(...)`.

**`async function measurePinSelection(project, missionA, missionB)` — the two-mission disambiguation measurement.** Enable the conductor over HTTP (`POST { project, enabled: true }`) so the state matches production. Use a stub invoke that records the pass rather than spawning a real node:
```ts
let invoked = 0; let lastSpec: any = null;
const stubInvoke = async (spec: any) => { invoked++; lastSpec = spec; return { ok: true, rateLimited: false, text: 'measured' } as any; };
```
Then, for each pin state, POST the pin over HTTP, call `runConductorPass(project, { invoke: stubInvoke })` in-process, and record the selected `result.missionId`:
- Pin `missionB` via HTTP → `runConductorPass` → `check` `result.reason === 'conducted'` **and** `result.missionId === missionB` (the pin, not the first-active A).
- Pin `missionA` via HTTP → `runConductorPass` → `check` `result.missionId === missionA` (pin swaps the driven mission).
- Clear pin (POST `targetMissionId: null`) via HTTP → `runConductorPass` → `check` `result.missionId === missionA` (unpinned falls back to first-active).
- Print a small measured summary line per state (`pinnedTo → selected`) so the harness output is a legible measurement, not just booleans.
Between the first two states the criteria fingerprint differs by mission, so no debounce interferes; if a same-mission re-pass is added, note it will `debounce` (expected).

**Cleanup (`finally`).** POST `targetMissionId: null` + `enabled: false` over HTTP to leave the deployed server clean, then `rmSync(project, { recursive: true, force: true })`. (The temp `watched_project` row is harmless and keyed to a now-deleted path; optionally leave a comment noting it.)

**Imports:** `mkdtempSync`, `rmSync`, `existsSync` from `node:fs`; `tmpdir` from `node:os`; `join` from `node:path`; `addWatchedProject` from `../src/services/supervisor-store`; `runConductorPass` from `../src/services/conductor-pass`; `forgeMission` from `../src/mcp/tools/mission-forge`.

### Acceptance criteria (positive, citable)
1. `scripts/smoke-conductor-pin-live.ts` defines `main` that health-gates on `${BASE}/api/health`, seeds two forged missions, runs both measurement phases, and `process.exit(fail === 0 ? 0 : 1)`.
2. `scripts/smoke-conductor-pin-live.ts` defines `measurePinRoundTrip`, which sets and reads the pin exclusively via `fetch` to `/api/supervisor/conductor` on the deployed server and asserts set/get/enable-preserves/clear.
3. `scripts/smoke-conductor-pin-live.ts` defines `measurePinSelection`, which POSTs each pin state over HTTP and calls `runConductorPass` with a stub `invoke`, asserting `result.missionId` equals the pinned mission (B, then A) and the first-active mission (A) when cleared.
4. The script imports `runConductorPass` from `../src/services/conductor-pass` and `forgeMission` from `../src/mcp/tools/mission-forge` and uses `addWatchedProject` to make the temp project watched before any POST.

### Out of scope
- No changes to `conductor-pass.ts`, `supervisor-store.ts`, or the route — this leaf only *measures* the existing pin path.
- Not added to the `bun test` runner (it is a live `bun run` smoke requiring a server, like the sibling `smoke-*-live.ts` scripts).
- Does not spawn a real conductor LLM node (stubbed `invoke`), so no `land_epic`/model cost.
